# Fix: Tab Identity & Persistence (Sub-agent links)

## Проблема
После перезапуска приложения панель `SubAgentBar` над терминалом Gemini оказывалась пустой, хотя связанные суб-агенты физически существовали в списке вкладок. Связи "родитель-ребенок" между терминалами разрушались.

### Симптомы
- Перезагрузка → суб-агенты пропали из `SubAgentBar`.
- Табы суб-агентов (Claude) "всплывали" в основном списке вкладок, вместо того чтобы быть скрытыми под Gemini.

## Root Cause: Tab ID Regeneration
Проблема заключалась в способе генерации идентификаторов вкладок при восстановлении из базы данных.

1. **Генерация:** Метод `createTab()` создавал ID по шаблону `projectId-tab-<tabCounter>`.
2. **Restore Cycle:** При запуске приложения `tabCounter` сбрасывался или инициализировался на основе *количества* сохраненных табов.
3. **ID Shift:** Если в прошлой сессии вкладки удалялись, порядковые номера в ID смещались. Например, сохраненный таб с ID `tab-5` при восстановлении мог получить ID `tab-10`.
4. **Broken Links:** Поле `parentTabId` в SQLite хранило старое значение (например, `tab-5`). Поскольку родительский таб теперь имел ID `tab-10`, фильтры в UI не находили совпадений, и связь разрывалась.

## Решение: Persist Original Tab ID
Архитектура изменена так, чтобы вкладки сохраняли свой уникальный ID на протяжении всей жизни проекта.

1. **Схема БД:** В таблицу `tabs` добавлена колонка `tab_id TEXT`. Это единственный источник истины для идентификации вкладок.
2. **Persistence:** При сохранении (`saveTabs`) в БД записывается текущий `tab.id`.
3. **Restore Logic:** При восстановлении метод `createTab()` принимает опциональный `restoreId`. Если он передан, генерация нового ID пропускается.
4. **Collision Protection:** Глобальный счетчик `tabCounter` при загрузке проекта инициализируется как `max(restored_tab_numbers) + 1`. Это предотвращает конфликт ID при создании новых вкладок после рестарта.

## Дополнительный фикс: Closure Trap (Link Provider)
Во время реализации кликабельных ID в терминале был исправлен классический баг замыканий в JS.

**Проблема:**
```javascript
let match;
while ((match = regex.exec(text)) !== null) {
  links.push({
    activate() { 
      // ❌ match здесь всегда равен null к моменту клика, 
      // так как цикл закончился и переменная переписана
      ipcRenderer.send('mcp:focus-task', match[0]); 
    }
  });
}
```

**Решение:**
Захват значения UUID в локальную константу внутри каждой итерации цикла.
```javascript
while ((m = re.exec(text)) !== null) {
  const uuid = m[0]; // ✅ Значение зафиксировано в скоупе итерации
  links.push({
    activate() { ipcRenderer.send('mcp:focus-task', uuid); }
  });
}
```

## Статус агентов после рестарта
**Персистентные поля суб-агентов (сохраняются в БД):**
- `mcp_task_id` — оригинальный UUID задачи из `delegate_to_claude`. Позволяет Gemini обращаться к агенту по тому же ID после рестарта.
- `claude_task_count` — счётчик выполненных итераций. SubAgentBar показывает реальное "N tasks" вместо 0.
- `parent_tab_id`, `claude_session_id`, `name`, `tab_id`, `mcp_task_started_at` — базовая идентичность.

**Runtime-only поля (НЕ персистятся, сбрасываются при рестарте):**
- `claudeAgentStatus` → `'done'` (PTY мёртв после рестарта)
- `claudeActive` → `false` (Claude CLI не запущен)
- `claudeBusy` — спиннер неактуален
- `interceptorState` — сбрасывается в `armed` при новом `continue_claude`
- `read_claude_history` — stateless, каждый вызов заново строит backtrace из JSONL. Нет серверного состояния для сброса.

**Сохранение имен:** Колонки `name` и `tab_id` обеспечивают восстановление кастомных заголовков суб-агентов (заданных через MCP), предотвращая их сброс к дефолтному `claude-sub`.
