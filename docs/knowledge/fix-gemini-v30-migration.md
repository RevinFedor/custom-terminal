# Fix: Gemini CLI v0.30 Migration (Project Resolution)

## Проблема
С версии 0.30 Gemini CLI изменил механизм идентификации проектов. Ранее папки чатов в `~/.gemini/tmp/` именовались на основе SHA256-хеша абсолютного пути к проекту. Теперь используются человекочитаемые **slug-и** (например, `noted-terminal`), а соответствие путей и slug-ов хранится в специальном индексном файле.

## Решение: Dynamic Slug Resolution
Система теперь использует двухэтапный поиск директории проекта:

1. **Source of Truth:** Чтение `~/.gemini/projects.json`. Это JSON-объект, где ключами являются slug-и, а значения содержат массив `directories`. Система ищет slug, у которого один из путей совпадает с текущим `CWD`.
2. **Fallback:** Если проект не найден в `projects.json` (например, старая версия CLI или проект еще не индексирован), система возвращается к расчету SHA256-хеша от пути.

### Почему это важно (Invisible Intent)
Без этого маппинга функции **Sniper** (автозахват сессий) и **Timeline** (отображение истории) перестают работать для новых проектов, так как ищут файлы в несуществующих хешированных папках.

## Влияние на подсистемы
- **Sniper Watcher:** Теперь динамически перезапускает `fs.watch` при обнаружении slug-а, так как путь к `chats/` изменился.
- **Session Manager:** При сохранении сессии в БД теперь записывается и slug, и хеш для обеспечения кросс-версионной совместимости.

## Связанные файлы
- `src/main/main.js` — функция `resolveGeminiProjectDir(cwd)`.
- `knowledge/fact-gemini-automation.md` — работа Sniper.
