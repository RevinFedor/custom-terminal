# Feature: Sub-Agent Lifecycle V2 (Recovery & Persistence)

## Intro
Система делегации задач (Gemini → Claude) перешла от модели «один таб — одна задача» (fire-and-forget) к модели «длительных сессий». Это позволяет Gemini повторно использовать существующих суб-агентов даже после полного перезапуска приложения.

## 1. Механизм обнаружения (Discovery)
Gemini CLI не имеет собственной памяти о созданных задачах после перезагрузки. Для восстановления контекста используется инструмент `list_sub_agents`.

### Алгоритм поиска (Hybrid Search):
1. **Memory:** Поиск в `mcpTasks` Map (актуально для текущей сессии Electron).
2. **SQLite Fallback:** Если Map пуст, система выполняет `SELECT` из таблицы `tabs`, фильтруя по `parent_tab_id` (ID текущего Gemini-таба).
3. **Synthetic IDs:** Для агентов, найденных в БД, генерируются временные `taskId` по шаблону `restored-<8_char_hash>`. 
   - Эти ID детерминированы и позволяют Gemini вызывать `continue_claude`, даже если он «забыл» оригинальный UUID задачи.

### Семантические состояния (list_sub_agents):
Вместо технических деталей (PTY alive), Gemini получает список с именами вкладок (например, `fix-auth`) и понятными статусами:
- **`ACTIVE`**: Claude запущен и готов к вводу или выполняет задачу.
- **`IDLE`**: Claude завершил предыдущую задачу и находится в шелле. `continue_claude` автоматически выполнит resume.
- **`STOPPED`**: Процесс PTY завершен. `continue_claude` создаст новый таб и восстановит сессию.

## 2. Авто-восстановление (The Zero-Friction Resume)
Инструмент `continue_claude(taskId, prompt)` обеспечивает бесшовную работу с «мертвыми» процессами. Система сама определяет, нужно ли запускать Claude CLI (`--resume`) или можно пастить промпт напрямую. Подробнее см. `knowledge/fix-mcp-v2-lifecycle-stability.md`.

### Сценарии выполнения:
- **PTY Alive:** Если процесс Claude ещё запущен, промпт отправляется мгновенно через `safePasteAndSubmit`.
- **PTY Dead (Post-restart):** 
    1. Система считывает `claude_session_id` из SQLite.
    2. В существующем табе автоматически запускается `claude --resume <sessionId>`.
    3. Используется стейт-машина **Handshake** для ожидания готовности промпта.
    4. Промпт вставляется автоматически сразу после готовности CLI.

## 3. Инспекция истории (read_claude_history)
Инструмент позволяет Gemini «вспоминать» ход мыслей и действия Claude. Чтобы не перегружать контекст Gemini длинными диалогами, чтение работает по **инкрементальной схеме**.

### Механика Watermark (deliveredTurns):
Сервер Noted Terminal автоматически отслеживает, какой объем истории Gemini уже видел.
1. **Первичная доставка:** При автоматической доставке первого ответа (`auto-delivery`) сервер ставит метку (например, `deliveredTurns = 3`).
2. **Инкрементальный запрос:** По умолчанию `read_claude_history` возвращает только **новые** сообщения, появившиеся после этой метки.
3. **Обновление метки:** После каждого успешного чтения ватермарка сдвигается вперед. Если новых сообщений нет, инструмент возвращает статус `(No new turns since last read)`.

### Параметры инструмента:
- **`taskId`**: Идентификатор задачи (поддерживаются `restored-` ID).
- **`detail`**: Уровень детализации (`summary`, `full`, `with_code`).
- **`from_beginning` (bool)**: Принудительный сброс ватермарки. Позволяет Gemini получить всю историю сессии с самого начала (например, если он потерял контекст).

### HTTP API:
Эндпоинт: `GET /claude-history/{taskId}?detail=...&from_beginning=...`
(НЕ `/history/` — этот эндпоинт не существует, запрос получит 404)

### Уровни детализации:
- **`summary`**: Только последний ответ Claude + `thinking`. Не возвращает промежуточные ходы.
- **`full` (Default)**: Все ходы: тексты + `thinking` + Action Labels (без кода). Формат: `--- Turn N/M ---`.
- **`with_code`**: Включает диффы Edit/Write и вывод Bash-команд. Read-результаты **не включаются** (`includeReading=false` hardcoded).

### Invisible Intent:
Инкрементальный подход избавляет Gemini от необходимости считать ходы (« turns ») вручную. Поскольку Gemini получает ответ суб-агента как обычный текст в терминале, он не знает его порядковый номер в JSONL. Сервер берет эту ответственность на себя, обеспечивая Gemini только «дельтой» свежей информации.

## 4. Протокол «Мета-доставки» ответов
Критически важно для понимания логики ИИ: результат выполнения задачи НЕ возвращается как прямой ответ MCP-инструмента.

### Flow:
1. Gemini вызывает `delegate_to_claude` / `continue_claude`.
2. Инструмент мгновенно возвращает `{ status: 'accepted', taskId: '...' }`. Это «освобождает» Gemini для другой работы.
3. Приложение Noted Terminal следит за PTY суб-агента через **Spinner Detection**.
4. Как только Claude заканчивает работу (состояние IDLE), приложение извлекает ответ из JSONL и **впрыскивает** его в поток ввода Gemini как пользовательское сообщение.
5. **Формат:** Сообщение всегда обернуто в теги `[Claude Sub-Agent Response]...[/Claude Sub-Agent Response]`. 
   - **Footer:** В конце ответа выводится строка метаданных: `Tab: <name> | Task ID: <id> | Model: <model> | Context: <%>`. Имя вкладки (Tab Name) помогает визуально связать текстовый ответ с конкретным агентом в многозадачном режиме.

## 4. Изоляция и Видимость
- **Hidden Tabs:** Суб-агенты всегда имеют `parentTabId` и скрыты из основного TabBar, чтобы не создавать визуальный шум.
- **Detach:** Пользователь может «отсоединить» суб-агента (ПКМ → Detach), превратив его в обычную вкладку. При этом связь в `mcpTasks` сохраняется, и Gemini всё ещё может отправлять туда команды.

### Итерации и счетчик задач
Чтобы Gemini понимал прогресс работы субагента, в систему введен инкрементальный счетчик `claudeTaskCount`.
- Обновляется при каждом переходе задачи в статус `done` или `error`.
- Отображается на чипе в `SubAgentBar` (например, "3 tasks"), позволяя пользователю видеть объем проделанной работы без открытия таба.
