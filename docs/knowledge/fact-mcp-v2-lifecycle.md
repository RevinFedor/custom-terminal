# Feature: Sub-Agent Lifecycle V2 (Recovery & Persistence)

## Intro
Система делегации задач (Gemini → Claude) перешла от модели «один таб — одна задача» (fire-and-forget) к модели «длительных сессий». Это позволяет Gemini повторно использовать существующих суб-агентов даже после полного перезапуска приложения.

## 1. Механизм обнаружения (Discovery)
Gemini CLI не имеет собственной памяти о созданных задачах после перезагрузки. Для восстановления контекста используется инструмент `list_sub_agents`.

### Алгоритм поиска (Hybrid Search):
1. **Memory:** Поиск в `mcpTasks` Map (актуально для текущей сессии Electron).
2. **SQLite Fallback:** Если Map пуст, система выполняет `SELECT` из таблицы `tabs`, фильтруя по `parent_tab_id` (ID текущего Gemini-таба).
3. **Persisted Task IDs:** Колонка `mcp_task_id` в таблице `tabs` хранит оригинальный UUID задачи, присвоенный при `delegate_to_claude`. После рестарта `list_sub_agents` возвращает эти реальные ID, а не синтетические.
   - **Fallback:** Если `mcp_task_id` отсутствует (старые табы до миграции), генерируется `restored-<full_tab_id>`.
4. **Claude Session ID Fallback:** При резолве `taskId` в инструменте `update_docs` добавлен поиск по колонке `claude_session_id`. 
   - **Мотивация:** В восстановленных сессиях Gemini часто берет UUID из футера ответа, который является ID сессии Claude, а не внутренним ID задачи. Это делает связку максимально устойчивой.

### Семантические состояния (list_sub_agents):
Вместо технических деталей (PTY alive), Gemini получает список с именами вкладок (например, `fix-auth`) и понятными статусами:
- **`ACTIVE`**: Claude запущен и готов к вводу или выполняет задачу.
- **`IDLE`**: Claude завершил предыдущую задачу и находится в шелле. `continue_claude` автоматически выполнит resume.
- **`STOPPED`**: Процесс PTY завершен. `continue_claude` создаст новый таб и восстановит сессию.

## 2. Авто-восстановление (The Zero-Friction Resume)
Система обеспечивает бесшовную работу с «мертвыми» процессами через три механизма:
1.  **`continue_claude(taskId, prompt)`**: Возобновляет существующую задачу.
2.  **`delegate_to_claude(..., session_id)`**: Создает нового субагента на базе старой сессии.
3.  **`close_sub_agent(taskId)`**: Финализирует жизненный цикл, закрывая таб и очищая память.

Подробнее см. `knowledge/fix-mcp-v2-lifecycle-stability.md`.

### Сценарии выполнения:
- **PTY Alive:** Если процесс Claude ещё запущен, промпт отправляется мгновенно через `safePasteAndSubmit`. Система принудительно переводит интерцептор в состояние **ARMED**, гарантируя доставку ответа.
- **PTY Dead (Post-restart / New Tab Resume):** 
    1. Система считывает `claude_session_id` (либо переданный в `delegate`, либо из SQLite).
    2. В табе автоматически запускается `claude --resume <sessionId>`.
    3. Используется стейт-машина **Handshake** для ожидания готовности промпта.
    4. Промпт вставляется автоматически сразу после готовности CLI. Интерцептор устанавливается в **ARMED**.

## 3. Инспекция истории (read_claude_history)
Инструмент позволяет Gemini «вспоминать» ход мыслей и действия Claude. Gemini сам контролирует объём чтения через параметр `last_n`. Никакого серверного состояния (watermark) нет.

### Параметры инструмента:
- **`taskId`**: Идентификатор задачи (поддерживаются `restored-` ID).
- **`detail`**: Уровень детализации (`summary`, `full`, `with_code`).
- **`last_n` (int)**: Количество последних ходов. По умолчанию `1` (только последний ход). `0` = вся сессия (использовать осторожно — JSONL может быть огромным).

### HTTP API:
Эндпоинт: `GET /claude-history/{taskId}?detail=...&last_n=...`
(НЕ `/history/` — этот эндпоинт не существует, запрос получит 404)

### Уровни детализации:
- **`summary`**: Только последний ответ Claude + `thinking`. Параметр `last_n` игнорируется.
- **`full` (Default)**: Ходы: тексты + `thinking` + Action Labels (без кода). Формат: `--- Turn N/M ---`.
- **`with_code`**: Включает диффы Edit/Write и вывод Bash-команд. Read-результаты **не включаются** (`includeReading=false` hardcoded).

### Дизайн (Stateless):
Нет скрытого серверного состояния. Backtrace-алгоритм каждый раз заново строит active branch из JSONL на диске. Gemini сам решает, сколько ходов ему нужно:
- Быстрая проверка: `detail="summary"` — последний ответ.
- Перечитать последний ход с кодом: `detail="with_code", last_n=1`.
- Контекст: `last_n=3`.
- Полный дамп: `last_n=0` (редко нужно).

### Почему не watermark (решение об отказе):
Ранняя версия использовала серверный `deliveredTurns` — скрытый счётчик, который автоматически сдвигался после каждого чтения. Это создавало три проблемы:
1. **Блокировка перечитывания:** Gemini получал ответ через auto-delivery (summary), затем хотел перечитать тот же ход с `detail="with_code"` — но watermark уже продвинулся, и сервер отвечал «нет новых ходов».
2. **Сброс при рестарте:** `deliveredTurns` жил только в памяти (`mcpTasks` Map). После перезапуска Gemini получал всю историю заново — потенциально огромный JSONL.
3. **Решение несуществующей проблемы:** Gemini как LLM сам способен решить, что ему читать. Серверу не нужно "помнить" за агента.

**Принцип:** Сервер — это stateless API поверх backtrace. Вся логика выбора объёма чтения — на стороне Gemini через `last_n`.

## 4. Протокол «Мета-доставки» ответов
Критически важно для понимания логики ИИ: результат выполнения задачи НЕ возвращается как прямой ответ MCP-инструмента.

### Flow:
1. Gemini вызывает `delegate_to_claude` / `continue_claude`.
2. Инструмент мгновенно возвращает `{ status: 'accepted', taskId: '...' }`. Это «освобождает» Gemini для другой работы.
3. Приложение Noted Terminal следит за PTY суб-агента через **Spinner Detection**.
4. Как только Claude заканчивает работу (состояние IDLE), приложение извлекает ответ из JSONL и **впрыскивает** его в поток ввода Gemini как пользовательское сообщение.
5. **Формат:** Сообщение всегда обернуто в теги `[Claude Sub-Agent Response]...[/Claude Sub-Agent Response]`.
   - **Footer:** В конце ответа выводится строка метаданных: `Tab: <name> | Task ID: <id> | Model: <model> | Context: <%>`. Имя вкладки (Tab Name) помогает визуально связать текстовый ответ с конкретным агентом в многозадачном режиме.
6. **Queue Persistence:** Если приложение закрывается, пока ответы находятся в очереди (`geminiResponseQueue`), они сохраняются в таблицу `app_state` (ключ `gemini_response_queue`) и автоматически восстанавливаются при следующем запуске.
## 4. Изоляция и Видимость
- **Hidden Tabs:** Суб-агенты всегда имеют `parentTabId` и скрыты из основного TabBar, чтобы не создавать визуальный шум.
- **Detach:** Пользователь может «отсоединить» суб-агента (ПКМ → Detach), превратив его в обычную вкладку. При этом связь в `mcpTasks` сохраняется, и Gemini всё ещё может отправлять туда команды.
- **Queue Indicator:** При наличии ответов в очереди в правой части `SubAgentBar` появляется фиолетовый индикатор «Queue: N» с пульсирующей точкой (соответствует цвету UUID в терминале). Тултип при наведении позволяет просмотреть превью ожидающих промптов.

### Синхронизация списка задач (mcpTasks Cleanup)
Для поддержания актуальности списка доступных агентов реализован механизм глубокой очистки:
- **Удаление при закрытии:** Когда таб суб-агента закрывается (через `close_sub_agent` или вручную), соответствующая запись полностью удаляется из Map `mcpTasks` в Main-процессе.
- **Предотвращение фантомов:** Это гарантирует, что `list_sub_agents` не будет возвращать закрытые вкладки со статусом `error`, избавляя Gemini от попыток писать в несуществующие терминалы.

### Итерации и счетчик задач
Чтобы Gemini понимал прогресс работы субагента, в систему введен инкрементальный счетчик `claudeTaskCount`.
- Обновляется при каждом переходе задачи в статус `done` или `error`.
- **Персистентный:** Сохраняется в колонке `claude_task_count` таблицы `tabs`. После рестарта SubAgentBar покажет реальное количество выполненных задач, а не 0.
- Отображается на чипе в `SubAgentBar` (например, "3 tasks"), позволяя пользователю видеть объем проделанной работы без открытия таба.

## 5. Timeout Management
Система ограничивает время выполнения одной задачи субагентом (лимит: 30 минут).
- **Reset on Activity:** Таймаут отсчитывается не от начала сессии, а от момента последнего промпта (`delegate_to_claude` или `continue_claude`). Каждый новый запрос обнуляет часы.
- **Persistence:** Время начала задачи (`mcp_task_started_at`), оригинальный `mcp_task_id` и счётчик `claude_task_count` сохраняются в БД, что обеспечивает полное восстановление контекста суб-агента после рестарта.
- **Safety:** При завершении работы, ошибке или закрытии вкладки таймер принудительно очищается. См. [`fix-mcp-timeout-stability.md`](fix-mcp-timeout-stability.md).
