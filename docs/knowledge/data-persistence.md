## 2. Ограничение Tailwind при работе с динамическими классами
**Файл-источник:** `fix-tailwind-dynamic-runtime.md`

### Суть
Tailwind — статический компилятор. Динамические строки типа `border-${c.color}-500` не попадают в итоговый CSS билд, так как сканер не видит полного имени класса.

### Решение
Использование **Inline Styles** для динамических значений (цвета табов, кастомные отступы).
```javascript
style={{ backgroundColor: c.bgColor }} // ✅ Всегда работает
```
Рекомендовано для любых элементов, цвет которых может менять пользователь.

## 3. Debounced Persistence (Disk & IPC Optimization)
**Файл-источник:** Сессия 2026-01-21, 2026-01-22

### Проблема
Частые IPC-вызовы к БД и запись в LocalStorage при каждом клике/переключении таба вызывали микро-фризы ("джиттер") интерфейса.

### Решение
Внедрение стратегии отложенного сохранения (**Debounce**):
- **Session State (SQLite):** Глобальное состояние (текущий `view`, идентификаторы сессий) теперь хранится в таблице `app_state` базы данных SQLite. Сохранение отложено на **300мс**.
- **Interrupted State:** Флаг `wasInterrupted` сохраняется в JSON-метаданных табов.
- **Tab Metadata (SQLite):** Сохранение через IPC `project:save-tabs` отложено на **500мс**.
- **Emergency Save:** При аварийном закрытии (`beforeunload`) используется синхронный `ipcRenderer.sendSync` для гарантии записи.

### Преимущество
Это разгружает шину IPC и дисковую подсистему, позволяя React мгновенно реагировать на ввод пользователя, а использование SQLite вместо LocalStorage повышает надежность хранения системных данных.

## Философия: От «Пути» к «Сущности»
Раньше проект в приложении определялся исключительно его путем на диске (path). Это было архитектурным тупиком: нельзя было открыть одну и ту же папку дважды с разными контекстами (наборами вкладок, чатами). База данных «схлопывала» данные при совпадении путей.

**Что изменилось:**
Теперь проект — это уникальная личность (инстанс) с собственным UUID. Путь (`path`) теперь является просто атрибутом (метаданными), как название или описание.

**Результат:**
Вы можете создать множество проектов, указывающих на одну и ту же физическую папку. У каждого будут свои терминалы, свои заметки и своя история AI-чатов. Они больше никогда не пересекаются.
Архитектура была изменена для поддержки множественных рабочих сред для одного и того же пути.

### 1. Миграция базы данных
В таблице `projects` базы данных SQLite был удален индекс `UNIQUE` с колонки `path`. Это позволило хранить несколько записей с одинаковым путем, но разными метаданными.

### 2. Генерация ID
Вместо хеширования пути теперь используется комбинация:
`base64(path).substring(0, 20)` + `timestamp` + `random_suffix`.
Это гарантирует уникальность ID даже при одновременном создании проектов.

### 3. Метод createProjectInstance
В `project-manager.js` добавлен метод `createProjectInstance`, который:
1. Проверяет наличие проектов по данному пути.
2. Генерирует новое имя с суффиксом (например, `my-project-1`, `my-project-2`), если проект уже существует.
3. Создает новую запись в БД и возвращает новый `projectId`.

## Результат
Пользователь может создавать неограниченное количество независимых рабочих сред из одной ированной директории (Bookmark). Каждая среда имеет свои табы, цвета и состояние AI-сессий.

# Experience: Tab Rename on Restart Fix

## Проблема
После перезапуска приложения вкладки, в которых была запущена AI-сессия (Claude/Gemini), сбрасывали свои кастомные имена и автоматически переименовывались в стандартные (`claude-01`, `gemini-01`).

## Причина
Логика автоматического переименования срабатывала при условии `!tab.commandType`. Поле `commandType` (тип запущенного процесса) хранилось только в оперативной памяти и не сохранялось в SQLite. После перезапуска значение становилось `undefined`, что триггерило "первый запуск" команды и принудительное переименование.

## Решение
1. **Схема БД:** В таблицу `tabs` добавлена колонка `command_type` (TEXT).
2. **Persistence:** Реализовано сохранение этого поля в методе `saveTabs` и его восстановление в `getProjectById`.
3. **Logic:** При возобновлении сессии `commandType` восстанавливается из БД, флаг `isFirstRun` становится `false`, и кастомное имя вкладки сохраняется.

## Урок для проекта
Любое свойство таба, влияющее на его визуальную логику или именование, должно быть частью персистентного состояния в БД.

# ОПЫТ: Сохранение истории терминала (Xterm Serialization)

## Проблема
В архитектуре приложения компонент `<Workspace />` размонтируется (unmount) при переходе пользователя на Dashboard. Поскольку инстансы `xterm.js` живут внутри компонентов терминала, они уничтожаются вместе с ними. 

**Последствия:**
- Визуальный буфер (текст в терминале) полностью очищается.
- Позиция прокрутки сбрасывается.
- Хотя PTY-процесс в Main-процессе остается живым и продолжает получать данные, пользователь видит пустой экран при возврате в воркспейс.

## Решение: Сериализация перед Unmount

Для решения проблемы используется `@xterm/addon-serialize`. 

### 1. Механизм сохранения
В компоненте `Terminal.tsx` в хуке `useEffect` (Cleanup function) вызывается метод сериализации:
```typescript
const serialized = serializeAddon.serialize();
if (serialized) {
  saveTerminalBuffer(tabId, serialized);
}
```
Данные сохраняются в `useWorkspaceStore` в `Map<string, string>`, где ключ — `tabId`.

### 2. Механизм восстановления
При повторном монтировании (mount) компонента, сразу после инициализации `xterm.js`, система проверяет наличие сохраненного буфера:
1. Если буфер есть, он записывается в терминал через `term.write(savedBuffer)`.
2. После записи буфер удаляется из Store (`clearTerminalBuffer`), чтобы не занимать память.
3. Сразу после восстановления "проигрывается" `pendingBuffer` (данные, которые пришли от PTY, пока терминал был размонтирован).

## Почему не `display: none` или `visibility: hidden`?
1. **Layout Jitter:** `xterm.js` чувствителен к размерам контейнера. Скрытие через CSS часто приводило к тому, что терминал считал свой размер равным 0x0, что ломало внутренние индексы строк.
2. **React Lifecycle:** Глобальная навигация реализована через смену `view` в стейте, что идеологически предполагает смену страниц (unmount старой, mount новой).

## Результат
Пользователь может свободно переключаться между списком проектов и открытым терминалом, не теряя контекст вывода и историю команд. Память расходуется экономно, так как сериализованная строка хранится только в момент "перелета" между экранами.

# ОПЫТ: Потеря данных PTY при unmount

## Проблема
В системе используется виртуализация вкладок: когда пользователь переходит на другой экран (например, Dashboard), компонент `Terminal.tsx` размонтируется. 
При unmount:
1. Текущий буфер сохраняется через `serializeAddon`.
2. Слушатель `ipcRenderer.on('terminal:data')` удаляется.

Однако процесс **PTY в Main процессе продолжает работать**. Если Claude (или любая команда) продолжает выдавать текст, пока таб не активен, эти данные **теряются**, так как их некому принять. При возврате на таб пользователь видит "дыру" в истории.

## Решение (Концепт)
Для полного устранения проблемы требуется глобальный менеджер буферов в Renderer процессе:
1. Глобальный слушатель `terminal:data` на уровне `App.tsx` или Store.
2. Аккумуляция данных в `Map<tabId, string>` для всех неактивных вкладок.
3. При mount терминала — конкатенация сериализованного буфера и накопленных "пропущенных" данных.

## Текущий статус
Проблема выявлена в ходе сессии и требует внедрения глобального слушателя в следующем цикле рефакторинга.

# Fact: Multi-Selection Model

## Обзор
В проекте реализована модель множественного выбора вкладок, позволяющая выполнять массовые действия (копирование ID, экспорт сессий, закрытие).

## Техническая реализация
- **State:** Массив `selectedTabIds` внутри `ProjectWorkspace` в `useWorkspaceStore.ts`.
- **Управление:**
    - `Shift + Click`: Выделение диапазона от текущей активной вкладки до кликнутой.
    - `Cmd/Ctrl + Click`: Добавление/удаление конкретной вкладки из выделения.
    - `Обычный клик`: Сброс выделения и выбор одной вкладки.
- **Визуализация:**
    - Выделенные вкладки получают пунктирную рамку или повышенную яркость фона.
    - **Критическое правило:** Цветная линия-индикатор (`borderTop`) отображается **ТОЛЬКО** на текущей активной вкладке, даже если выделено несколько.

## Массовые действия
При выделении более одной вкладки интерфейс адаптируется:
1.  **ProjectToolbar:** Кнопка "Copy Session" превращается в "Copy X Sessions".
2.  **Context Menu:** Скрываются инструменты редактирования (Rename, Color), остаются только массовые (Close Tabs, Copy IDs).
3.  **Drag-and-Drop:** Все выделенные вкладки переносятся группой при перетаскивании любой из них на другой проект. См. [`knowledge/data-persistence.md`](data-persistence.md).

## Правила сброса
Выделение является временным состоянием и сбрасывается в следующих случаях:
- Клик по любой области вне `TabBar` (терминал, сайдбар).
- Создание новой вкладки через `Cmd+T`.
- Обычный клик (без Shift/Cmd) по любой вкладке (выделяет только её).

# ОПЫТ: Реализация Multi-Tab Drag-and-Drop (Cross-Project)

## Проблема
Стандартная реализация DnD в `TabBar` позволяла перемещать только одну вкладку за раз. При наличии множественного выделения пользователю приходилось перетаскивать каждый таб по отдельности, что неудобно при реорганизации рабочих пространств.

## Решение: Групповой перенос

Для реализации массового переноса вкладок на другие проекты была изменена структура данных и визуализация процесса.

### 1. Расширение DragData
Тип данных перетаскивания теперь включает список всех выделенных вкладок:
```typescript
type DragData = {
  type: 'TAB';
  id: string; // ID таба, за который схватились
  projectId: string;
  selectedTabIds: string[]; // Все выделенные табы в этом проекте
};
```
Если пользователь тянет таб, который **не входит** в текущее выделение, `selectedTabIds` содержит только этот один таб.

### 2. Кастомный Drag Preview (Бейдж "+N")
Для визуальной индикации того, что переносится группа, используется `setCustomNativeDragPreview` из Pragmatic DnD.
- Если переносится >1 вкладки, стандартный "призрак" заменяется на компактный блок.
- Блок содержит имя текущего таба и синий бейдж с количеством дополнительных вкладок (например, `tab-1 +2`).

### 3. Batch Move Logic (Store)
В `useWorkspaceStore` реализована функция `moveTabsToProject`.
- **Атомарность:** Все вкладки удаляются из исходного проекта и добавляются в целевой в одном цикле.
- **PTY Preservation:** Сами объекты вкладок (включая PID и состояние PTY) не меняются, меняется только их "прописка" в БД и памяти.
- **Cleanup:** После переноса в исходном проекте очищается выделение и корректируется `activeTabId`.

### 4. Визуальный фидбек
Во время перетаскивания всем вкладкам из группы `selectedTabIds` устанавливается `opacity: 0.5`. Это дает пользователю четкое понимание того, какие именно элементы покинут текущий проект при отпускании кнопки мыши.

## 4. Placeholder Paths for Empty Projects
В старых версиях базы данных (Production) колонка `path` в таблице `projects` имеет ограничение `UNIQUE`. При создании "пустого" проекта без выбранной папки возникал конфликт при попытке вставить вторую пустую строку.

**Решение:**
- Пустые проекты создаются с путем `__unset__new_project_{projectId}`.
- **UI Filter:** В `ProjectCard` и `Workspace` пути, начинающиеся с `__unset__`, отображаются как "No path".
- **CWD Fallback:** В `useWorkspaceStore` логика создания таба проверяет путь. Если он является плейсхолдером, `cwd` терминала сбрасывается на `process.env.HOME`.

## 5. Tab Navigation History (LRU Persistence)
Для обеспечения удобной навигации в `useWorkspaceStore` внедрен стек `tabHistory`.
- **Логика:** При каждом `switchTab` ID таба перемещается в конец массива (стека).
- **Navigation:** При закрытии текущего таба система ищет в стеке ближайший доступный (не удаленный) таб. Это позволяет пользователю возвращаться "назад" по истории своих переключений, а не просто к соседнему табу.
- **Limit:** Стек ограничен 20 последними записями для экономии памяти.

# ОПЫТ: Сброс scroll position в скрытом терминале

## Проблема
При переключении между вкладками терминала, если в скрытом табе продолжают поступать данные (например, Claude Code печатает), при возврате на этот таб происходит "дёргание" — терминал показывает верх буфера вместо ожидаемой позиции.

**Корневая причина:** Баг xterm.js — при записи данных в терминал с `visibility: hidden`, `viewportY` сбрасывается в 0, хотя `baseY` (размер буфера) продолжает расти.

Логи демонстрируют проблему:
```
viewportY= 466 baseY= 466 isAtBottom= true   ← до записи
viewportY= 0   baseY= 471 isAtBottom= false  ← после записи в скрытый терминал
```

## Решение
Вместо сохранения только `scrollPosition`, дополнительно сохраняем флаг `wasAtBottom`:

```typescript
const savedScrollPosition = useRef<number | null>(null);
const wasAtBottom = useRef<boolean>(true);

// При деактивации таба:
savedScrollPosition.current = currentViewportY;
wasAtBottom.current = currentViewportY >= currentBaseY;

// При активации таба:
if (wasAtBottom) {
  term.scrollToBottom();  // Игнорируем сброшенный viewportY
} else {
  term.scrollToLine(savedScrollPosition);  // Пользователь был проскроллен вверх
}
```

## Критическое правило
При восстановлении scroll position после переключения табов **нельзя полагаться на текущий `viewportY`** — он может быть сброшен в 0 из-за бага xterm.js. Используй сохранённый флаг `wasAtBottom` для определения нужного поведения.
