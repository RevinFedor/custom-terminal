# Fix: Gemini-to-Claude Delegation Handshake & Prompt Splitting

## Проблема: Склеивание команд (Slash-Command Concatenation)

### Симптомы
При делегации задачи из Gemini в Claude (через MCP инструмент `delegate_to_claude`) пользователь видел ошибку:
`Model 'haiku
[текст промпта...]' not found`.

### Причина (Invisible Intent)
Gemini CLI часто формирует комплексный ответ, где первой строкой идет смена модели, а затем сам промпт:
```
/model haiku
Проведи исследование проекта...
```
Стандартный механизм **Handshake** в приложении вставлял весь этот текст целиком через Bracketed Paste. Claude Code парсит содержимое сразу после закрывающего тега вставки. Поскольку текст шел единым блоком, Claude воспринимал всё, что следует за `/model`, как название модели, включая переносы строк и основной текст задачи.

## Решение: sendHandshakePrompt (Sequential Dispatch)

Был внедрен механизм разделения (splitting) промпта на этапе рукопожатия:
1. **Парсинг:** Первая строка проверяется на наличие префикса `/`.
2. **Атомарность:** Каждая slash-команда отправляется как отдельное событие вставки с немедленным ``.
3. **Prompt Return Wait:** После каждой команды система ждет появления символа `⏵` или `>` в PTY-потоке. 
   - **Почему это важно:** Без ожидания следующая часть промпта (сама задача) могла "улететь" в Claude, пока тот еще переключает модель или обрабатывает предыдущую команду, что приводило к потере данных.
4. **Body Injection:** Основное тело промпта отправляется только после подтверждения готовности Claude.

## Отброшенные подходы

### 1. Прямой `terminal:input` IPC (Bracketed Paste)
Попытка отправлять промпт через стандартный канал пользовательского ввода в свежесозданные Gemini-табы провалилась. 
- **Результат:** Текст появлялся в терминале, но не сабмитился, либо Gemini игнорировал вставку.
- **Причина:** На этапе инициализации alternate buffer в Gemini TUI обработка вставки нестабильна.
- **Решение:** Переход на `gemini:send-command`, который использует `safePasteAndSubmit` с подтверждением через sync-маркеры.

## Неочевидные связи

### Command Type Sync
При программном запуске Gemini через IPC (`gemini:spawn-with-watcher`) тип команды таба (`commandType`) автоматически не обновлялся, так как основной перехват логики завязан на нажатие клавиши Enter пользователем.
- **Следствие:** Sniper ловил сессию, но `SubAgentBar` не появлялся, так как он реактивно зависит от `commandType === 'gemini'`.
- **Фикс:** В `handleGeminiSessionDetected` добавлена принудительная установка `commandType`, что "пробуждает" UI-панели управления сессией.
