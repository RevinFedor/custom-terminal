# Fix: OSC 7777 Escape Carryover (PTY Chunking)

## Problem: Broken Escape Sequences
При использовании Claude Code в терминале иногда наблюдался визуальный баг: вместо цветного промпта пользователь видел сырые ANSI-параметры как текст (например, `38;2;153;153;153m❯`).

### Симптомы
- Начало промпта превращается в текст типа `38;2;153;153;153m`.
- Цвет промпта (обычно серый) теряется, текст становится дефолтного цвета.
- Проблема проявляется случайно (в 1-5% случаев), чаще при высокой нагрузке на систему.
- Функциональность Timeline (click-to-scroll) при этом продолжает работать корректно.

## Root Cause: PTY Kernel Chunking
Проблема вызвана физическим ограничением уровня ядра (PTY TTY line discipline).

1. **Разрыв последовательности:** Ядро PTY может разбить вывод приложения на произвольные чанки (обычно 1024 байта). Если чанк заканчивается *внутри* многобайтовой escape-последовательности (например, Truecolor CSI: `\x1b[38;2;...m`), она разрывается на два куска.
2. **Инъекция Middleware:** Наш PTY Middleware (в Main процессе) отслеживает промпт для вставки маркера OSC 7777. Когда он видит начало промпта в начале нового чанка (например, `38;2;...m❯`), он вставляет (prepend) `\x1b]7777;prompt:<uuid>\x07` ПЕРЕД этими данными.
3. **Аборт парсера xterm.js:** xterm.js получает: `[Хвост чанка 1: \x1b[]` + `[Наш OSC 7777]` + `[Начало чанка 2: 38;2;...m❯]`.
   - Встретив `\x1b]` (OSC) внутри незавершенного CSI, парсер xterm.js аварийно завершает (абортит) предыдущую последовательность.
   - Оставшаяся часть последовательности (`38;2;...m`) больше не распознается как команда и рендерится как обычный текст.

## Решение: Escape Carryover (Буферизация хвостов)
В Middleware реализован механизм детекции и склейки разорванных последовательностей.

### Алгоритм (src/main/main.js)
1. **Детекция:** Перед обработкой чанка Middleware проверяет его конец на наличие "незавершенного" Escape или CSI (ищет байт `\x1b` или последовательность `\x1b[` без закрывающей буквы).
2. **Буферизация:** Если хвост обнаружен, он отрезается от текущего чанка и сохраняется в `carryoverBuffer`.
3. **Склейка:** При поступлении следующего чанка `carryoverBuffer` приклеивается в его начало.
4. **Инъекция:** Только ПОСЛЕ склейки выполняется логика проверки промпта и вставки OSC 7777.

```javascript
// Псевдокод логики в main.js
let carryover = '';

pty.onData((chunk) => {
  const fullData = carryover + chunk;
  carryover = '';

  // Проверка на разрыв в конце fullData
  const lastEsc = fullData.lastIndexOf('\x1b');
  if (lastEsc !== -1) {
    const tail = fullData.slice(lastEsc);
    if (isUnfinishedSequence(tail)) {
      fullData = fullData.slice(0, lastEsc);
      carryover = tail;
    }
  }

  // Теперь безопасно впрыскивать OSC 7777
  const processed = injectMarkers(fullData);
  renderer.send(processed);
});
```

## Отброшенные подходы
- **Инъекция после (Append):** Мы не знаем, где именно в предыдущем чанке началась последовательность, поэтому не можем вставить маркер сразу после неё.
- **Задержка (Delay):** Ожидание следующего чанка ломает детерминизм и может привести к рассинхрону маркеров с реальным временем прихода данных.

## Верификация
Для проверки реализован Headless-тест `auto/sandbox/test-osc7777-escape-corruption.js`, который:
1. Генерирует 38 сценариев разрыва (включая разрывы на `\x1b`, `\x1b[`, `\x1b[38`, `\x1b[38;2;` и т.д.).
2. Симулирует подачу данных в Middleware кусками.
3. Проверяет, что xterm.js (через `@xterm/headless`) не зафиксировал текстовых артефактов в буфере.
