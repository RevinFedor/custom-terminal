# Fix: Gemini Response Queue & Input Detection

### Симптомы
- Ответы суб-агентов Claude вставляются в Gemini прямо поверх набираемого пользователем текста, ломая промпт.
- При получении нескольких ответов одновременно они «дерутся» за фокус терминала, вызывая мешанину из текста.
- Очередь не очищается автоматически, даже если пользователь удалил весь свой текст вручную (Backspace).

---

## 1. Проблема: Нестабильность Alternate Buffer
Изначально планировалось определять наличие текста в инпуте Gemini путем парсинга буфера xterm.js.
**Почему отказались:**
- Gemini CLI использует **Alternate Screen Buffer** и Ink TUI, который перерисовывает весь экран при каждом событии.
- Поиск «строки ввода» по координатам или текстовым якорям крайне хрупок: координаты «плывут» при изменении размера окна, а якоря зависят от темы оформления и длины промпта.

## 2. Решение: Счётчик символов (Main Process)
Вместо визуального анализа внедрен перехват на уровне `terminal:input` в Main-процессе.

### Механика трекинга:
1. **`geminiInputCharCount`:** Map, хранящая количество «грязных» символов для каждого таба.
2. **Инкремент:** Любой печатный символ увеличивает счётчик.
3. **Декремент (Backspace):** Система ловит коды `\x7f` и `\x08`. Это позволяет точно знать, когда пользователь очистил поле ввода вручную.
4. **Сброс (Reset):** Счётчик обнуляется при нажатии Enter, Ctrl+C, Ctrl+U или переходе Gemini в состояние BUSY (факт отправки).

### Авто-доставка (Auto-delivery):
Как только счётчик достигает **0** и Gemini находится в состоянии **IDLE**, система автоматически вызывает `processGeminiQueue`. Это обеспечивает мгновенную вставку ответа, как только пользователь стёр свой черновик.

## 3. Принудительная вставка (Force Flush)
Для случаев, когда пользователь хочет проигнорировать свой ввод, добавлена кнопка **«Send now»**.

### Логика прерывания:
При нажатии кнопки система выполняет серию команд для очистки PTY:
1. Сброс `geminiInputCharCount` в ноль.
2. Отправка `Ctrl+C` (прерывание текущего режима ввода в Gemini).
3. Отправка `Ctrl+A + Ctrl+K` (гарантированная очистка строки, если Gemini остался в INSERT моде).
4. Мгновенный вызов `deliverToGeminiImmediate`.

**Почему кнопка одна на всю очередь:** В Gemini CLI ответы обрабатываются строго последовательно. Нельзя вставить второй ответ, пока первый не был «проглочен» и обработан (BUSY -> IDLE). Поэтому кнопка всегда относится к «следующему в очереди» сообщению.

## 4. Invisible Intent: Hover Bridge
Для предотвращения закрытия тултипа очереди при движении мыши от бейджа к списку сообщений используется **Invisible Bridge Pattern**. Между элементами оставлен микро-зазор с прозрачным оверлеем, удерживающим hover-состояние.
