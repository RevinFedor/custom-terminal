# Fix: MCP Sub-Agent Stability (Claude vs PTY)

## 1. Концепция: Claude CLI Active ≠ PTY Alive

### Проблема
После перезапуска приложения или восстановления вкладок из SQLite возникала ситуация «Ложного присутствия». 
- **PTY Alive:** Терминал монтируется и автоматически спавнит `zsh`/`bash`. Процесс в ОС существует.
- **Claude CLI Dead:** Исполняемый файл Клода еще не запущен внутри этого шелла. Он ждет либо нажатия кнопки «Resume» пользователем, либо команды от MCP.

Старые методы проверки (`terminals.has(tabId)`) считали такие табы «живыми», что приводило к попыткам вставить промпт прямо в голый шелл.

### Решение: Единственный источник истины (`claudeCliActive`)
Введен специальный Map `claudeCliActive` в Main-процессе.
- **True:** Только после реального запуска команды `claude` (через делегацию или авто-resume).
- **False:** При выходе Клода обратно в шелл или завершении PTY.
- **IPC:** Статус транслируется в Renderer для отображения ●/◌ индикаторов.

## 2. Platform Bug: OSC 133 D Race (Shell Init)

### Симптомы
Сразу после делегации чип субагента на мгновение становился зеленым, а затем навсегда серым, хотя Claude продолжал работать.

### Причина
Протокол **OSC 133** (Shell Integration) в `zsh` отправляет последовательность `133;D` (Command Finished) при инициализации интерактивного промпта. Это происходит **ДО** того, как Claude успевает перехватить управление PTY. Система видела сигнал завершения команды и ошибочно сбрасывала флаг `claudeCliActive`.

### Решение: Handshake Guard
Механизм очистки статуса в Main-процессе теперь игнорирует сигналы `133;D`, если для этого таба активен стейт-машина **Handshake** (`claudeState` или `claudePendingPrompt`). Статус считается валидным только после того, как Клод реально "проглотил" промпт.

## 3. Архитектура: Атомарный `continue_claude`

### Логика "Самоисцеления"
Инструменту `continue_claude` больше не нужно, чтобы Claude CLI был запущен заранее.
1. **Проверка:** Если PTY жив, но `claudeCliActive` — false.
2. **Phase 3b (Auto-resume):** Система сама пишет `claude --resume <id>` в шелл.
3. **Handshake:** Дожидается готовности промпта.
4. **Injection:** Вставляет задачу.

Это делает оркестрацию прозрачной для Gemini: он просто "говорит" с агентом, а терминал берет на себя всю техническую работу по оживлению процесса.

## 4. PID Cache (Баг "Исчезновения" MANGO)

### Проблема
Прямой обход дерева процессов через `ps` нестабилен. Если дерево процессов меняется между вызовами (например, шелл перезапускается), связь PPID → TabID могла теряться, и агент исчезал из списка `list_sub_agents`.

### Решение
Внедрен кеш `ppidToGeminiTab`. Один раз успешно сопоставленный PID "родителя" (Gemini) жестко привязывается к ID вкладки до конца жизни процесса MCP-сервера.

## 5. PTY Respawn (Self-healing)

### Проблема: Недоступность LAZY_INIT
В приложении используется механизм ленивой инициализации терминалов (`LAZY_INIT`). Процесс PTY создается только тогда, когда пользователь реально открывает вкладку.
**Ловушка:** Вкладки суб-агентов (`parentTabId`) по умолчанию скрыты. Пользователь никогда не кликает по ним напрямую. Если PTY суб-агента упал (например, `exitCode=1` после случайного Ctrl+C), он никогда не будет пересоздан автоматически, так как механизм `LAZY_INIT` не сработает.

### Решение: Принудительный Respawn
Для инструмента `continue_claude` реализована цепочка команд "самоисцеления" при обнаружении мертвого PTY:
1. **Main Process:** Обнаруживает отсутствие PTY в `terminals` Map.
2. **IPC Request:** Отправляет сигнал `mcp:respawn-sub-agent-pty` в Renderer.
3. **Renderer (App.tsx):** Вызывает метод `terminal:create`, который заново спавнит PTY с нужным CWD.
4. **Guard:** В хендлер `terminal:create` добавлен замок, предотвращающий создание дублирующих процессов при одновременном срабатывании `LAZY_INIT` и `mcp:respawn`.
5. **Retry Loop:** Инструмент `continue_claude` ждет появления нового PTY в Map, после чего выполняет `claude --resume`.

Это обеспечивает 100% доступность суб-агентов даже при нестабильной работе шелла или аварийных завершениях.
