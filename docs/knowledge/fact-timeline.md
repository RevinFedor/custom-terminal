# Feature: Claude History Timeline

## Intro
Таймлайн — это вертикальная панель справа от терминала, отображающая хронологию сообщений в текущей сессии Claude Code. Позволяет быстро визуализировать прогресс и перемещаться к конкретным точкам диалога.

- **User Flow**
1. Пользователь запускает `claude` в терминале.
2. Справа появляется полоса с точками. Каждая точка — сообщение пользователя.
3. При наведении на точку появляется всплывающее окно с превью текста промпта.
4. **Клик на точку (Контекстный):**
   - Если панель истории **закрыта**: терминал скроллится к месту промпта.
   - Если панель истории **открыта**: происходит переход к соответствующему сообщению внутри панели истории. Клик по самому терминалу в этом режиме игнорируется.
   - **Исключение (Compact/Continued):** Записи типов `compact` и `continued` (Context Overflow) не имеют физической позиции в терминале. Клик по их маркерам на Timeline работает **только** при открытой History Panel, выполняя скролл внутри ленты.

## 2. Behavior Specs
- **Visibility (Range-based):** Timeline отображается всегда, когда в текущем табе есть активная сессия AI.
    - **Logic:** Вместо поиска текста промпта во вьюпорте используется принцип «владения блоком». Каждое сообщение пользователя "владеет" диапазоном строк от своей позиции до начала следующего сообщения.
    - **Indicator:** Точка превращается в черточку (Viewport Dash), если вьюпорт терминала пересекается с диапазоном строк этого сообщения. Это гарантирует, что таймлайн всегда показывает активную точку, даже если пользователь скроллит длинный ответ AI, а сам промпт уехал за экран.
    - **Response-Only Indicator:** Если вьюпорт пересекает диапазон entry, но позиция самого промпта (`markerRow`) находится **выше** верхней границы вьюпорта — значит пользователь видит только ответ AI. В этом случае черточка отображается приглушённым серым (`rgba(255,255,255,0.35)`) без glow, вместо яркого белого (`rgba(255,255,255,0.75)` + glow). Размер черточки одинаковый (10px). Определение: `blockStart < viewport.top`. Работает для обоих движков (Claude OSC markers, Gemini text positions).
    - **Invisible Intent (Психология фокуса):** Приглушение индикатора необходимо, чтобы пользователь мгновенно понимал: на экране сейчас только результат работы ИИ, а сам вопрос находится выше. Это снижает когническую нагрузку при быстром скролле длинных диалогов.
    - **Shield Logic:** Маркеры сжатия (`compact`) служат приоритетной границей. Если сразу после сжатия происходит техническая смена ID сессии, система блокирует создание ложного `FORK` маркера. См. [`fix-compact-logic.md`](fix-compact-logic.md).
    - **Граница актуальности (Old History Zone):** Область истории до последнего сжатия (Compact) или сброса контекста (Plan Mode) визуально отделяется от активного диалога.
        - **Визуал:** Вся зона получает приглушённый красный фон (`rgba(239, 68, 68, 0.06)`).
        - **Клик:** Клик по точкам в этой зоне заблокирован (cursor: default), так как этого контента гарантированно нет в терминальном буфере Claude.
        - **Unified Boundary Marker:** Точка последней границы помечается белой линией на всю ширину (100%) Timeline, создавая четкий разрыв между "архивным" и текущим контекстом.
    - **Индикация доступности контента (Hollow Rings):** Позволяет мгновенно отличить сообщения, которые можно просмотреть в терминале, от удалённых.
        - **Заполненная точка:** Контент найден в буфере терминала (reachable). Клик выполнит скролл.
        - **Полое кольцо (Ring):** Контент отсутствует в буфере (unreachable) из-за очистки скроллбека или принадлежности к Old History Zone. При клике появится красный крестик.
    - **Bounds:** Timeline и панель истории ограничены верхней границей терминала. Они не заходят в область табов проектов или терминалов.
    - **Tab Persistence:** Состояние панели истории (открыта/закрыта) и её размер сохраняются индивидуально для каждой вкладки терминала. При переключении вкладок состояние восстанавливается.
    - **Persistent Mount:** Timeline никогда не удаляется из DOM при переключении на Home, чтобы не вызывать скачков верстки. Он скрывается через проп `isVisible`, сохраняя свою ширину (32px).
    - **Active State:** Если процесс Claude/Gemini запущен, Timeline имеет стандартную яркость.
    - **Inactive State:** Если агент не запущен, Timeline переходит в "приглушенный" режим (фон `rgba(40,40,40,0.4)`, прозрачность `0.6`). Это позволяет просматривать историю даже после завершения работы агента.
- **Docs Updated (Зелёные точки):** Автоматические маркеры правок документации.
    - **Logic:** Система сканирует записи `assistant` в JSONL на наличие вызовов инструментов `Edit`/`Write` для файлов в `docs/` или `CLAUDE.md`.
    - **Grouping:** Несколько правок из одного ответа ассистента группируются в одну зелёную точку.
    - **Visual:** Зелёный цвет (`#4ade80`). Не имеют позиции в терминале (как компакт-записи).
- **Note-on-Dot Integration:** При выборе позиции заметки «на самой точке» (`position: 'dot'`) визуальная полоска-индикатор скрывается.
    - **Consolidation:** Содержимое заметки встраивается инлайном в верхнюю часть основного тултипа промпта (фиолетовый блок с иконкой `StickyNote`). Это устраняет необходимость в отдельном Cmd-ховере для просмотра заметки.
- **Edit-Range Persistence on Tab Switch:** При смене `sessionId` editRangeState (ready/done/loading) сохраняется в module-level Map (`_savedEditRangeMap`), остальные overlay-состояния (rewind, copying, tooltip, contextMenu, unreachableIndices) сбрасываются. Module-level Map (не useRef) — переживает unmount компонента при смене проекта или showTimeline=null.
    - **Отброшенные подходы (5+ итераций):** (1) Promote loading→ready с пустым текстом — пользователь видел пустой диалог вместо загрузки. (2) Auto-refetch через executeRangeAction при restore — бесконечный loop (executeRangeAction меняла state → триггерила useEffect → повторный вызов). (3) Clean Slate (всё сбрасывать) — потеря готового результата Gemini при переключении. (4) Re-fetch зависящий от entries — stale entries от другого таба вызывали ошибку "план мод". (5) useRef для saved Map — умирал при unmount (смена проекта), данные терялись.
    - **sessionIdRef Guard:** Async fetch (`fetchEditRangeCompact`) захватывает `sessionId` prop через closure на момент вызова. После tab switch closure видит СТАРЫЙ sessionId, но `setEditRangeState` обновляет state текущего (нового) таба. AbortController ненадёжен — HTTP response может быть буферизирован до abort signal (race condition, подтверждено логами: response приходил через 4с после abort). Решение: `sessionIdRef` (ref-зеркало prop, обновляется на каждом рендере) + обязательный `expectedSessionId` параметр. Guard проверяет `sessionIdRef.current !== expectedSessionId` после await — даже если abort не сработал, state не запишется на wrong tab, а сохранится в Map для правильного таба.
    - **Cleanup closure trap:** useEffect cleanup захватывает `editRangeState` из момента создания effect (stale). Ref-зеркало `editRangeStateRef` обновляется через wrapper `setEditRangeState` — cleanup читает ref (всегда актуальный).
    - **Unreachable reset:** При смене sessionId unreachableIndices/visibleEntryIndices/responseOnlyIndices сбрасываются в пустой Set — stale индексы от предыдущего таба + терминал нового таба = всё unreachable = красный Timeline.
- **Backtrace Filtering:** Отображение только активной ветки диалога. Сообщения, отменённые через Escape или Undo, скрываются из Timeline автоматически.
- **Auto-Refresh:** Timeline обновляется автоматически каждые **2 секунды**, чтобы отражать актуальное состояние сессии. Обновление загружает только данные из JSONL/JSON — поиск по буферу терминала для обновления самих точек не выполняется.
    - **Performance Guard (`isVisible`):** Поллинг данных и запросы к SQLite выполняются только тогда, когда Timeline реально виден пользователю. Это предотвращает лишнюю нагрузку на диск и БД.
  - **rAF-debounced Scroll Check (Feb 2026):** Функция `checkScrollPosition` (обновляющая кнопку "scroll down" и viewport rect в registry) вызывается только один раз за кадр анимации (requestAnimationFrame). Ink differential renderer генерирует множество `onWriteParsed` событий в секунду — без дебаунса каждое событие вызывает пересчет Timeline и React re-render. rAF-дебаунс коллапсирует N событий в одно, предотвращая рывки при прокрутке длинных ответов.

### Sizing & Scrolling
Entries используют `flex: '1 0 auto'` с минимальной высотой (20px обычные, 14px дочерние агенты, 22px tree mode).
- **Мало entries:** Растягиваются равномерно на всю высоту контейнера (flex-grow: 1).
- **Много entries:** minHeight сохраняется, контейнер начинает скроллиться.
- **Разделители** (fork/plan markers): `flex-shrink: 0`, фиксированная высота 8px — не растягиваются.

### Layout Constraints (The Flex-Height Trap)
Для корректной работы скролла внутри сложной flex-сетки воркспейса используется специальный CSS-хак.
- **Корневая причина:** Стандартный `height: 100%` на flex-child (скролл-контейнере) часто не резолвится Chromium, если родитель не имеет фиксированной высоты. Это приводило к тому, что контейнер Timeline растягивался на всю высоту контента вместо появления скроллбара.
- **Решение:** Использование `position: absolute; inset: 0` для внутреннего скролл-контейнера. Это гарантирует, что контейнер будет ограничен физическими размерами родительской 32px-полосы, заставляя `overflow-y: auto` работать корректно.

### Визуальный фидбек и навигация
- **Click Feedback:** При клике на точку мгновенно появляется вращающийся спиннер (`animate-spin`). 
- **Error Handling:** Если текст сообщения не найден в буфере (например, из-за очистки истории), на месте точки на 1.2с появляется красный крестик.
- **Tooltip Sync:** Тултип отображается вплотную к левому краю таймлайна с мягким белым свечением (`shadow-white`) для лучшей читаемости.
- **Auto-Refresh & Debounce:** Обновление данных происходит каждые 2с. Проверка видимости во вьюпорте (трансформация в черточку) выполняется при скролле с дебаунсом 100мс. Проверка доступности в полном буфере (красный фон) — с дебаунсом 500мс.
- **Reachability Guard (`canDetermineReachability`):** `checkReachability()` помечает entries как unreachable (`pos < 0`) только если терминал существует в `terminalRegistry`. Если терминала нет (startup до монтирования, HMR) — ничего не красится. Когда терминал есть, но текст entries отсутствует в буфере (процесс закрыт, буфер пуст) — все entries корректно становятся красными. Прежний guard `hasAnyPosition` (красить только если хоть одна entry найдена) ошибочно блокировал покраснение при закрытом Gemini (0 из 106 found → guard = false → ничего не красное).
- **Boundary Gate Fix:** OSC 7777 injection в main.js гейтится на `claudeSpinnerBusy.has(tabId)` (не `bridgeKnownSessions`). См. [`fix-boundary-bridge-race.md`](fix-boundary-bridge-race.md).
- **Opaque Dots:** Все точки используют непрозрачные (solid) цвета. Это необходимо, чтобы они полностью перекрывали центральную вертикальную линию (Central Axis), исключая визуальный мусор и «просвечивание» полосы сквозь индикаторы.
- **Transition Policy (Targeted, Not `all`):** Точки анимируют width, height, border-radius (0.15s), цвет и glow (0.2s). Это безопасно: максимальный размер точки (10px) всегда меньше `minHeight` сегмента (20px/14px), поэтому flex reflow не происходит. **Нельзя** использовать `transition: all` — это захватит свойства сегмента (padding, margin) и заставит Chromium пересчитывать flex-layout при наведении, вызывая дрожание всех entries.

### Кастомный скроллбар (In-strip Indicator)
Для Timeline реализован кастомный индикатор скролла, заменяющий системный скроллбар.
- **Почему не нативный:** Нативный скроллбар браузера (даже стилизованный через webkit) конфликтовал с расположенным рядом `Resizer`. Клик по скроллбару часто интерпретировался системой как начало ресайза боковой панели.
- **Реализация:** Тонкий (3px) ярко-белый индикатор внутри 32px-полосы Timeline.
- **Изоляция:** Индикатор имеет `pointer-events: none`. Пользователь скроллит Timeline вращением колеса мыши над его областью, а физические клики «пролетают» сквозь индикатор, не мешая работе соседнего ресайзера.
- **Direct DOM Update (No Re-render):** Позиция индикатора и видимость Scroll-to-Bottom Arrow обновляются через `ref.style` напрямую в scroll handler, без React state (`useState`). Прежний подход (`setScrollThumb()`, `setShowScrollDown()` на каждый scroll event) вызывал полный re-render тяжёлого Timeline компонента (~2300 строк JSX) на каждый тик колёсика — контент скроллился рывками, обновляясь только при остановке мыши.

### Scroll-to-Bottom Arrow
Фиолетовая стрелка (`ChevronDown`, цвет `#a78bfa`) в нижней части Timeline.
- **Условие показа:** Появляется только когда ниже видимой области есть reachable (не-красные) entries. Проверка выполняется через `querySelectorAll('[data-segment]')` на контейнере `scrollRef`.
- **Действие:** Плавный скролл (`behavior: 'smooth'`) scrollRef контейнера до конца.
- **Синхронизация с History Panel:** При нажатии кнопки, если открыта боковая панель истории, система отправляет кастомное событие `history-panel:scroll-to-bottom`. History Panel слушает его и также выполняет скролл к последним сообщениям.
- **Детекция:** Встроена в существующий `updateThumb()` scroll handler (без дополнительных event listeners). При `isAtBottom` (scrollTop >= maxScroll - 5) стрелка скрывается.
- **Motivation:** В длинных сессиях (50+ entries) пользователь может случайно проскроллить вверх и не знать, что ниже есть актуальные (не-красные) данные.
- **Invisible Intent:** Использование `querySelectorAll('[data-segment]')` вместо прямого обхода `children` необходимо из-за наличия в `scrollRef` технических обёрток (relative wrappers) и анимационных оверлеев, которые не являются сегментами данных.

### Token Count Indicator (Cmd+Hover)
При зажатом Cmd и открытом tooltip в footer отображается оценка токенов: `⚡ ↑342 ↓2.3K`.
- **↑ (User message):** Эвристика `Math.ceil(content.length / 3.5)`. Оффлайн-токенизатора для Claude 3+ не существует — Anthropic не опубликовали. Пакет `@anthropic-ai/tokenizer` (npm) — только Claude 1/2. Для кода/JSON разница ±10%, для русского текста хуже (~1 токен ≈ 2 символа).
- **↓ (Response):** Точные `output_tokens` из `message.usage` в JSONL assistant entries.
- **Ловушка: несколько assistant entries за ход.** Claude пишет отдельный assistant entry на каждый шаг (thinking → tool_use → text). Каждый имеет свой `message.usage`. Нужно **суммировать** `output_tokens` по всем assistant entries до следующего user.
- **Ловушка: `input_tokens` ≠ размер сообщения.** Поле `input_tokens` в usage — только uncached часть контекста. Полный контекст = `input_tokens + cache_read_input_tokens + cache_creation_input_tokens` (может быть 50K+ для длинных сессий). Поэтому для ↑ используется эвристика от длины текста, а не API-данные.

### Auto-Scroll при продолжении сессии
При нажатии «Продолжить» (claude-c / gemini-c) Timeline автоматически прокручивается к последним entries. Также срабатывает при первой загрузке entries с новым sessionId (запуск приложения). Не sticky-scroll — при периодическом обновлении (каждые 2с) позиция скролла не меняется, чтобы не мешать ручному просмотру истории.
- **Invisible Intent:** Без auto-scroll пользователь при продолжении длинной сессии видит верх Timeline с красными (unreachable) entries, хотя актуальные записи внизу. SessionId при «Продолжить» не меняется, поэтому обычный watch на sessionId не срабатывает — нужен отдельный сигнал от кнопки.

### Click-to-Scroll (Deterministic Navigation)

#### Claude Code (OSC Markers)
Для Claude Code навигация является детерминированной и работает через систему **PTY-маркеров**.
- **Механизм:** При клике система вызывает `terminalRegistry.scrollToEntry(uuid)`. Это мгновенный прыжок $O(1)$ к физической строке в буфере, зафиксированной в момент прихода данных от Claude.
- **Точность:** Для вычисления положения оверлеев (выделение, анимация отката) используются реальные DOM-координаты сегментов (`offsetTop`/`offsetHeight` через `segmentRefs`), а не проценты (`% / length`). Это необходимо, так как наличие fork-маркеров и flex-grow делает высоту каждого сегмента индивидуальной.
- **Escape Carryover:** В PTY Middleware реализована защита от разрыва escape-последовательностей при вставке маркеров (PTY chunking). См. `knowledge/fix-osc7777-escape-carryover.md`.
- **Fallback:** Если маркер отсутствует (например, для самого первого сообщения сессии или после очистки скроллбека), система откатывается к поиску через `SearchAddon`.

#### Scrollable Absolute Overlays
При реализации анимаций Timeline (желтая пульсация выбранного диапазона, фиолетовая пульсация активной точки) оверлеи должны физически двигаться вместе с контентом.

- **Проблема:** Стандартный `position: absolute` внутри контейнера с `overflow: auto` привязывается к экранным координатам (вьюпорту), а не к скроллируемому содержимому. При прокрутке оверлеи «залипают» на месте.
- **Решение:** Все оверлеи обернуты в **inner relative wrapper** — промежуточный контейнер с `position: relative`, который охватывает весь список записей. Это делает оверлеи частью скроллируемого слоя, обеспечивая их стабильное положение относительно точек при любом уровне прокрутки.

#### Gemini CLI (Heuristic Search)
Поскольку Gemini работает в Alternate Buffer, маркеры xterm.js там не работают. Навигация использует многострочный поиск по тексту с алгоритмами повышения устойчивости:

- **Truncation Fallback:** Gemini TUI обрезает длинные сообщения в отображении (показывает только начало с префиксом `> `). Если multi-line поиск по полному тексту из JSON проваливается, система принимает match по одной первой строке при условии `firstLine.length >= 15`. Конкретная длина обрезки зависит от версии Gemini CLI и не зашита в коде.
- **Anchor Fallback:** Для предотвращения ложных срабатываний поиск каждой следующей записи начинается после позиции предыдущей (`startAfterRow`). Если поиск с якорем не дал результатов (якорь был ошибочным), система автоматически повторяет поиск по всему буферу.
- **Near-separators Filtering:** При генерации поисковых ключей (`getSearchLines`) система игнорирует не только чистые разделители (`────`), но и строки с "шумом" (например, `────╯`), где частота одного символа составляет $>80\%$. Это предотвращает использование декораций TUI как уникальных якорей.
- **Improved Thresholds:** Порог `isIsolatedShort` для коротких сообщений (типа "продолжай") увеличен с $+10$ до $+25$ символов, чтобы учитывать отступы и декорации, добавляемые Gemini TUI.

### Click-to-Scroll (Context-aware)
- При клике на точку вызывается `terminalRegistry.searchAndScrollToNth`.
- **Валидация контекста:** Поиск игнорирует вхождения текста внутри других сообщений. Скролл происходит только к строкам, которые начинаются с символов промпта (`❯`, `⏵`) или валидного отступа (для вставленного кода). Это синхронизирует логику клика с визуальным отображением точек.
- **Prompt Detection Fix:** Символ `>` исключен из списка промптов, так как он часто встречается в markdown-цитатах ответов Claude, вызывая ложные срабатывания.
- Предварительного кеширования позиций (marker binding) нет — поиск выполняется по требованию при клике.
- Используется таймер 250ms для различения одинарного и двойного клика.
- **Search Robustness (Viewport Matching):** Для поиска используется только первые **50 символов** сообщения (вместо 80). Это гарантирует, что ключ поиска не пересечет границу переноса строки (line wrap), который в Ink TUI часто случается раньше из-за отступов. Из поискового запроса удаляются специальные символы.

### Fork & Plan Mode Markers (визуальные разделители)
Тонкие (8px) полоски между entries, обозначающие границы сессий.
- **Fork marker** (синий `#3b82f6`): Появляется после entry, которая является последней в snapshot'е форка. Означает: «здесь сессия была разветвлена».
- **Plan mode marker** (бирюзовый `#48968c`): Появляется между entries с разными `sessionId`. Означает: «здесь был Clear Context / Plan Mode».
- **Взаимоисключающие:** Если entry имеет fork marker, plan mode boundary не рендерится (и наоборот).
- **Beginning markers:** Если fork/plan произошёл до первой entry (пустой snapshot), маркер рендерится в самом верху Timeline.
- **Viewport Dash:** Маркеры реагируют на viewport — при пересечении вьюпорта с диапазоном entry, dash становится шире (10px vs 12px).
- **Без красного фона:** Маркеры-разделители не имеют собственного unreachable-фона — красный показывается только на segment-div'ах entries выше и ниже.
- **Soft-delete:** Удаление fork marker через UI ставит `hidden=1` в БД. Hidden маркеры не рендерят полоску, но snapshot'ы остаются для `forkSnapshotSets` (подавление ложных plan mode границ в старых форках).
- **CMD+hover tooltip:** Fork marker поддерживает CMD+наведение (паттерн заметок). Показывает session ID источника + кнопку "Удалить маркер". Позиционирование: midpoint между `segmentRefs[N].bottom` и `segmentRefs[N+1].top` (fork strip рендерится МЕЖДУ entries, а не на entry).

### Smart Tooltip
- При наведении на точку появляется превью сообщения.
- **Layout:** Текст промпта (центральный блок) + нижний бар `[Expand]  timestamp  ............  [Copy]`.
- **Scrolling:** Текст промпта имеет собственный скролл-контейнер с `scrollbar-thin` (3px). Сообщения не обрезаются (`line-clamp` удалён), полный текст всегда доступен через скролл даже в свёрнутом виде.
- **Copy Animation:** Кнопка копирования меняет текст на «Скопировано» и иконку на галочку при успехе (на 1.2с).
- **Resize:** Пользователь может изменять ширину тултипа, потянув за левую границу. Ширина сохраняется в рамках сессии. См. [`fix-timeline-tooltip-stability.md`](fix-timeline-tooltip-stability.md).
- Использует "Invisible Bridge" для плавного наведения (см. `knowledge/fact-react-patterns.md`).
- Для сообщений типа `continued` тултип показывает статус "Context Overflow Recovery".

**Запуск режима выделения:**
- **Двойной клик** по точке, ИЛИ
- **ПКМ** → "Начать копирование"

**Работа в режиме выделения:**

- **Одинарный клик по другой точке** → popup-меню с выбором: "Копировать" или "Редактировать".

### Edit Range (компактирование диапазона)
Полный пайплайн: выделение → Gemini compact → JSONL edit → Claude restart.

**Ограничение:** Редактирование работает **только в пределах текущего сегмента** (после последнего plan mode / compact boundary). Попытка редактирования через границы показывает тост с ошибкой. Мотивация: JSONL-файлы split по plan mode (каждый Clear Context = новый файл), edit-range работает с одним файлом.

**Flow:**
1. Выбор диапазона → "Редактировать" → боковая панель `EditRangePanel`.
2. Содержимое диапазона отправляется в Gemini API для генерации compact-summary.
3. Пользователь может отредактировать compact или запросить уточнения через чат в панели.
4. "Применить": exit Claude (DangerZone Ctrl+C) → `claude:edit-range` IPC → wait prompt-ready → restart Claude.
5. `claude:edit-range`: backtrace → remove range entries → insert compact entry → relink parentUuid → atomic write.

**Пост-промпт (▶):** Split-button рядом с "Редактировать" и "Откатиться" — ▶ раскрывает inline textarea для дополнительной инструкции Gemini. Текст добавляется как `ПРИОРИТЕТНАЯ ИНСТРУКЦИЯ` перед основным rewind prompt. Мотивация: пользователь хочет направить фокус compact-сводки (например "сфокусируйся на коммитах, игнорируй обсуждение дизайна" или "перечисли только изменённые файлы"). Без пост-промпта единственный способ повлиять — редактировать результат вручную в EditRangePanel. Работает одинаково для Edit-range и Rewind.

**Прерывание и ручной ввод:**
- Правый клик на розовом loading overlay → контекстное меню: "Прервать" (AbortController.abort + открыть панель с пустым текстом) или "Отмена" (abort + закрыть). Мотивация: пользователь может не ждать Gemini и сразу вставить свой compact-текст.
- Кнопки ✏️ (Edit) и 📋 (Copy) под каждым ответом Gemini в EditRangePanel — inline-редактирование ответа Gemini (textarea замена, Esc отмена, Сохранить → `setMessages()`). Пустой ответ (после "Прервать") показывает подсказку "нажмите ✏ чтобы написать свой текст".

**Невидимый контекст (отброшенные подходы):**

**Архитектурная дивергенция: наш Backtrace vs Claude DAG.**
Наш Timeline backtrace: "последняя физическая запись в файле → идём назад по parentUuid". Claude `--resume`: "newest leaf by timestamp в DAG → идём назад по parentUuid". Это ДВА РАЗНЫХ алгоритма. Timeline может показывать корректную ветку, а Claude при resume пойдёт по другой (мёртвой) ветке — если у мёртвой ветки timestamp новее. Все правила ниже вытекают из этой дивергенции.

- **Compact timestamp в конце файла: `new Date().toISOString()`**, не `entryBefore.timestamp`. Если compact — последний entry в файле (нет entryAfter), его timestamp ОБЯЗАН быть newest. При использовании `entryBefore.timestamp` два entry получают идентичный timestamp → Claude не может однозначно определить newest leaf → выбирает assistant (завершённый turn) → игнорирует compact user entry.
- **Compact timestamp в середине файла:** ОБЯЗАН быть старше entries после него. Иначе Claude при resume строит DAG и выбирает compact как active leaf → Claude форкает от compact, игнорируя entries после.
- **Compact entry sessionId — зависит от позиции:**
    - **В середине файла (entryAfter exists):** `entryAfter.sessionId`. Entries вокруг compact могут иметь sessionId отличный от таба (после clear context / plan mode). Если compact получит sessionId таба, а entryAfter имеет другой → Timeline покажет ложный "Clear Context" маркер. Подтверждено в production: rewind внутри одного сегмента создал compact с `sid=ffdd5e0f` (таб), а entries после имели `sid=185346fb` (after clear context) → ложный boundary.
    - **В конце файла (нет entryAfter):** sessionId таба (IPC параметр). Claude `--resume` ищет newest leaf → sessionId обязан совпадать с тем что Claude использует. В форках entries имеют оригинальный sessionId (например `2fa76efe`), а Claude --resume использует fork sessionId (`e713a1a9`). Несовпадение → DAG fork → compact становится мёртвой веткой.
- **Orphan cleanup при edit-range:** Rewind создаёт мёртвые ветки (entries с parentUuid → удалённые записи). Edit-range удаляет часть active branch, но orphan'ы от rewind'ов оставались. Orphan'ы c newer timestamp → Claude resume подбирает orphan (chain=1) → 0% context. Решение: cascade-удаление orphan entries перед записью файла. Проходит несколько итераций (дети orphan'ов тоже orphan'ы). **КРИТИЧНО: entryAfter.parentUuid ОБЯЗАН быть перелинкован на compact entry ДО orphan cleanup.** Без этого entryAfter указывает на удалённую запись → orphan cleanup помечает его orphaned → cascade через ВСЮ оставшуюся цепочку → удаляет все entries после edit range. Подтверждено в production: пользователь выделил 4 записи, потерял 1481 (включая ~40 реальных разговоров). Также `compactUuid` должен быть добавлен в `survivingUuids` до orphan cleanup (compact entry ещё не в allRecords, но на него уже ссылается entryAfter).
- **Forward-expand ловушка (tool_result + system + task-notification):** После user-сообщения Claude отвечает цепочкой `assistant → tool_use → tool_result(type:user) → assistant → ...`. Forward-expand обязан проходить через: tool_result entries (`type: 'user'`, `content[].type === 'tool_result'`), system entries (turn_duration и другие metadata между turns), task-notification entries (`type: 'user'`, content начинается с `<task-notification>` — автоматические сообщения от sub-agent'ов). Без прохода через system/task-notification: forward-expand останавливается на system boundary, оставляя "призраков" — ответы Claude на task-notification от фоновых агентов, которые пользователь видит как мусор между compact и своим следующим сообщением (подтверждено: 13 ghost entries в production). Останавливаться ТОЛЬКО на настоящем user-сообщении (текстовый content, не tool_result, не `<task-notification>`).
- **`userType: 'external'` убивает контекст Claude CLI.** Claude CLI при `--resume` НЕ загружает в контекст записи с `userType: 'external'` — считает их "чужими". Если файл был создан через fork (copyFileSync) от сессии где записи имели эту метку (например prefilled session), а потом edit-range перезаписал файл — ВСЕ записи сохраняют `userType: 'external'` → Claude загружает 4% вместо 25%. Фикс: edit-range при перезаписи удаляет `userType: 'external'` со всех surviving entries. Метка нужна только на нашем compact entry.

### Rewind (Откатиться)
Автоматизация процесса возврата к выбранному сообщению через TUI-меню истории Claude Code.

**Визуальное исправление: Scroll Stabilization (Mar 2026)**
При откате Claude TUI выполняет интенсивную перерисовку (полное очищение буфера + загрузка истории), что вызывает множество скролл-событий и "прыжки" вьюпорта. Оверлеи Timeline (желтая пульсация выбранного диапазона, фиолетовая пульсация активной точки) раньше "залипали" в viewport при скролле, т.к. были привязаны к абсолютным координатам экрана.

**Решение:** Все оверлеи (анимационные span'ы) теперь находятся внутри `relative`-контейнера, который охватывает весь скроллируемый flex-список entries. Таким образом, при скролле контейнера оверлеи физически движутся вместе с контентом, а не остаются на месте.

**User Flow:**
1. **ПКМ по точке** → "Откатиться".
2. **Фаза 1 (Компакт):** Система копирует все сообщения от выбранного до конца сессии (`claude:copy-range`), отправляет их в Gemini с промптом "Rewind (Откат)" для получения краткого резюме.
3. **Фаза 2 (Откат):** Вызывается IPC `claude:open-history-menu`. Процесс имитирует `Esc`+`Esc` для открытия меню Claude, находит нужную запись по префиксу текста (Text-matching), навигирует стрелками и подтверждает выбор (`Enter`).
4. **Фаза 3 (Вставка):** После перезапуска Claude система очищает инпут (`Ctrl+C`), вставляет сжатый контекст через `safePasteAndSubmit` (программный paste: чанки < 1024 байт + sync marker verification + автоматический `\r`). Это **программная** операция — текст идёт напрямую в PTY, минуя xterm.js. См. `knowledge/fact-fact-terminal-core.md`.

**Scroll Stabilization (Авто-скролл):**
После отката Claude TUI часто вызывает "прыжки" скролла к началу буфера из-за интенсивной перерисовки. 
- Система автоматически подписывается на событие `onScroll` терминала после подтверждения отката.
- Используется **debounce 200ms**: каждое изменение скролла сбрасывает таймер.
- Когда скролл стабилизируется (не меняется 200мс) → вызывается `scrollToBottom()`.
- Также предусмотрен safety-фоллбэк на 5 секунд.

**Навигация (RGB Visual Search & Safety Guard):**
Для выбора записи в TUI-меню используется **Визуальный Поиск** с защитой.
- **Алгоритм:** Нажимает `Arrow Up` и анализирует сырой вывод PTY на предмет RGB-цвета выделения.
- **Нормализация:** Текст цели и текст на экране очищаются (`.trim()`) перед сравнением, чтобы игнорировать разницу в отступах.
- **Safety Guard:** Если после всех попыток текст не найден, операция **прерывается с ошибкой** (меню закрывается через `Escape`). Это предотвращает случайный выбор неверной записи (например, сброс в начало сессии).
- Детали реализации: `knowledge/fix-rewind-navigation.md`.

**Визуальная обратная связь (пульсация):**
Во время процесса точка и диапазон пульсируют (`animate-pulse`), отражая текущую фазу:
- **Янтарный:** Компактизация (Gemini работает).
- **Фиолетовый:** Навигация в TUI (Откат).
- **Зелёный (1.2с):** Успешная вставка и завершение.

**Три фазы визуальной обратной связи (Range Copy):**

| Фаза | Стейт | Визуал |
|-------|-------|--------|
| Выделение | `selectionStartId` | Статичный синий индикатор |
| Загрузка | `copyingRange` | Мигающий синий (`animate-pulse`) |
| Успех | `copiedRange` (800ms) | Зелёный flash с glow |

**IPC:** `claude:copy-range` — использует `resolveSessionChain` + Backtrace (тот же unified pipeline, что и `claude:get-timeline`). Это гарантирует корректную работу при копировании через границы Plan Mode.

### Context Menu (ПКМ)
- **Позиционирование:** Конкретное меню открывается справа-посередине от курсора (`x = cursor + 4px`, `y = cursor - height/2`). Высота меню вычисляется динамически на основе количества кнопок.
- **Auto-close:** Меню автоматически закрывается при уходе курсора с его области (`onMouseLeave`).
- **Portal Rule:** Кнопки в контекстном меню (Portal) обязательно используют `e.stopPropagation()` — см. `knowledge/fact-react-patterns.md`.
- **Button Status:** Во время отката кнопка "Откатиться" заменяется на статус выполнения ("Компакт...", "Откат...", "Вставка...") и блокируется.

### Sub-Agent Orchestration View (Gemini only)
При работе с оркестратором Gemini → Claude, Timeline поддерживает визуализацию работы суб-агентов.

**Phase 1 — Visual Differentiation:**
- Записи `[Claude Sub-Agent Response]` отображаются **индиго** цветом (`#818cf8`) вместо белого.
- Записи `[Claude Sub-Agent Timeout]` отображаются **красным** (`#ef4444`).
- Метаданные агента (имя, taskId) парсятся из footer ответа: `Tab: <name> | Task ID: <id> | Model: <model>`.
- **Footer Variants:** Footer может начинаться с `Tab:`, `Task ID:` или `Model:` (без `Tab:`/`Task ID:`). Если `Tab:` отсутствует, имя формируется из модели: `Claude (<model>)`.
- **Closing Tag Guard:** `parseSubAgentMeta()` требует наличие И открывающего `[Claude Sub-Agent Response]` И закрывающего `[/Claude Sub-Agent Response]`. Это предотвращает ложные срабатывания когда пользователь упоминает тег в тексте промпта.
- Backend: `parseSubAgentMeta()` в `gemini-data.js`.

**Phase 2 — Collapsible Groups:**
- **Раунды оркестрации:** ВСЕ последовательные sub-agent entries схлопываются в одну ноду (один "раунд оркестрации"), независимо от TaskID или имени агента.
- **Motivation:** В реальном workflow Gemini часто вызывает разных агентов последовательно (Research → Implementation → Test). Группировка всех ответов между двумя промптами пользователя в одну ноду значительно упрощает навигацию по Timeline, сокращая количество точек с 50+ до 10-15.
- **Collapsed:** одна индиго-точка с badge `×N` справа. Клик → expand.
- **Expanded:** toggle-бар сверху (горизонтальная индиго-линия 8×2px, клик → collapse) + ВСЕ entries группы как children (indent, уменьшенный dot 3px, кликабельные для навигации).
- **Dual-role header:** Первый entry группы (`isGroupHeader: true && isGroupChild: true`) — при collapsed работает как toggle, при expanded рендерится как child entry + отдельный toggle-бар сверху.
- **Tooltip Guard:** Badge `×N agents` в tooltip показывается только когда группа **collapsed** (`!expandedGroups.has(gi.groupId)`). При expanded каждый child entry показывает собственный tooltip без group badge.
- Группировка: `computeGroups()` возвращает `Map<number, GroupInfo>`. Собирает уникальные имена агентов (`Set<string>`). `agentName` = имя если один агент, `"N agents"` если несколько, `"Claude"` если unnamed.
- Visibility агрегируется для collapsed headers (any child visible → header visible).
- Состояние: `expandedGroups: Set<string>`, сбрасывается при смене сессии.

**Phase 3 — Tree View:**
- **Абсолютное наложение:** При активации режима дерева Timeline расширяется с 32px до 160px **влево**, используя `position: absolute`.
- **Motivation (Stability):** Использование абсолютного позиционирования поверх терминала (с `backdrop-filter: blur(8px)`) предотвращает Layout Shift. Терминал не меняет свои размеры, что критично для стабильности Canvas-рендерера и предотвращения лишних `SIGWINCH` сигналов в PTY (которые могли бы вызвать перерисовку TUI).
- **Labels:** В tree mode каждая нода показывает текстовый label (имя агента или превью промпта).
- **Визуал:** Вертикальные линии-коннекторы для sub-agent записей (CSS absolute positioned). Центральная ось перемещается влево (left: 4px).
- Tooltip `right` позиция учитывает ширину tree mode.
- Состояние: `treeMode: boolean`, персистится в `UIStore` (per-tab).
- **Trigger:** Кнопка переключения (иконка GitBranch) вынесена в `InfoPanel` (сайдбар) рядом с кнопкой "AI SESSION".

## Code Map
- `Timeline.tsx`: UI компонент (полоса, точки, портал для тултипов, range copy, sub-agent groups, tree view, scroll-down arrow).
- `Timeline.tsx` → `computePositions()`: Claude: OSC 7777 prompt boundaries → text fallback (`buildPositionIndex`). Gemini: text search.
- `Timeline.tsx` → `checkReachability()`: Двухпроходная система:
  1. **Regular entries** (`user`): unreachable если `index <= lastBoundaryIdx` ИЛИ `position < 0` (при `canDetermineReachability`).
  2. **Special entries** (`compact`, `continued`, `plan`): наследуют unreachable от следующей regular entry.
  - **`lastBoundaryIdx`**: Индекс последнего `compact`/`plan` entry. Compact и Plan Mode **стирают терминал** Claude — все entries до них гарантированно unreachable.
  - **Guard:** `canDetermineReachability = !!terminalRegistry.get(tabId)` — если терминал не существует (startup, HMR), ничего не красится. Если терминал есть — `pos < 0` → unreachable.
  - **Fork/Plan marker strips** (8px разделители между entries): НЕ имеют собственного красного фона — unreachable индикация только на segment-div'ах entries.
- `gemini-data.js` (`gemini:get-timeline`): Парсинг Gemini JSON + `parseSubAgentMeta()` для извлечения метаданных агентов.
- `ipc/claude-data.js` (`claude:get-timeline`): Парсинг и фильтрация JSONL. Детекция типа `continued`.
- `main.js` (~line 3190): OSC 7777 boundary injection state machine. Gate: `claudeSpinnerBusy.has(tabId)`.
- `terminalRegistry.ts`: Методы `searchAndScroll`, `getVisibleText` (чтение вьюпорта), `getPromptBoundaryLines`, `buildPositionIndex`.
- **Gemini Truncation Fallback:** Gemini TUI обрезает длинные сообщения пользователя (~137 символов). `getSearchLines()` извлекает до 3 строк из полного контента JSONL, но 2-я/3-я строки могут отсутствовать в буфере. `buildPositionIndex` и `scrollToTextInBuffer` используют truncation fallback: если первая строка >= 30 chars и найдена в буфере — match принимается даже без подтверждения остальных строк. Порог 30 chars гарантирует уникальность (для коротких первых строк вроде `[Claude Sub-Agent Response]` по-прежнему требуется multi-line match).
