## Mutual Panel Exclusion (History ↔ Tree Mode)

### Концепция
Когда несколько тяжелых боковых панелей могут отображаться одновременно, они конкурируют за:
- Визуальное пространство на экране
- Пропускную способность Main Thread при рендере

При открытии одной панели другая автоматически сворачивается, предотвращая перегруженность интерфейса и замедление.

### Реализация (History ↔ Tree Mode)
- **History Panel** (полная история сессии): Содержит список всех записей с текстом, фильтрацией, поиском. Рендеринг 200+ элементов занимает значительные ресурсы.
- **Tree Mode** (расширенная Timeline): Расширение Timeline с 32px до 160px для отображения текстовых меток (имена агентов, превью промптов, группы sub-agent оркестрации).

**Правило:**
```
При открытии History Panel → collapseTreeMode()
При активации Tree Mode → closeHistoryPanel()
```

**Обоснование:**
- Оба компонента рендерятся в левой области рядом с терминалом.
- Одновременно открытые — создают горизонтальный "бутерброд" из трех слоев (Tree + History + Terminal).
- Пользователь редко одновременно нуждается в обоих. History используется для глубокого поиска, Tree — для быстрой ориентации в структуре сообщений.

При скрытии одной панели остается видимым ровно один компонент боковой панели, предотвращая Layout Shift и конкуренцию за Main Thread.

## Compositor-Thread Animations (GPU Acceleration)

### Проблема
При загрузке тяжелых списков (200+ элементов) React блокирует Main Thread на 1-2+ секунды для reconciliation. В это время **все анимации на Main Thread замораживаются** — спиннеры, переходы, hover-эффекты становятся неподвижны. Пользователь видит "зависший" интерфейс.

### Решение: GPU-слой для анимаций
Некритичные анимации (спиннеры, лоадеры, вращения) переводятся на **Compositor-слой** браузера, который работает независимо от Main Thread.

### Реализация

**CSS:**
```css
.spinner {
  will-change: transform;
  animation: spin 1s linear infinite;
}

@keyframes spin {
  from { transform: rotate(0deg); }
  to { transform: rotate(360deg); }
}
```

**Механизм:**
1. **`will-change: transform`**: Браузер создает отдельный compositing layer для элемента с классом `.spinner`.
2. **GPU Rendering:** Анимация `transform: rotate()` обрабатывается на GPU, а не на Main Thread.
3. **Independence:** Даже когда JavaScript занят рендером DOM и парсингом JSONL, спиннер продолжает крутиться с полной частотой 60fps.

### Результат
- Пользователь видит **плавный**, постоянно вращающийся спиннер.
- UI кажется отзывчивым, даже если основной поток занят.
- Создается иллюзия параллельной работы, снижая когнитивную нагрузку и повышая доверие к приложению.

### Применение
- Timeline скролл-спиннер
- History Panel лоадер
- Modal dialogs лоадеры
- Sub-agent orchestration progress indicators

### Важно: Не все анимации
Не все анимации должны быть на GPU:
- **Интерактивные** (hover, click feedback) остаются на Main Thread для немедленного response на события.
- **Некритичные** (фоновые спиннеры, progress) — на GPU для стабильности.

## Toast Layout & Copy Persistence (API Results)
**Новый стандарт (2026-03)** для отображения результатов API-запросов в Toast уведомлениях.

### Layout
```
[✕] [Icon] [Message] [copy]
```

- **Кнопка закрытия (✕):** Слева (было: справа). Закрывает уведомление.
- **Иконка:** Визуальный индикатор типа (успех/ошибка/info).
- **Сообщение:** Основной текст результата API.
- **Кнопка Copy:** Справа. Повторное копирование результата API из уведомления.

### Motivation
Кнопка `copy` позволяет пользователю снова забрать результат API, не засоряя основной буфер обмена при случайном копировании другого текста. Toast автоматически исчезает через 8 сек, но результат остается в памяти Toast-компонента.

## Unified AI Launch (Split-Buttons)
Для минимизации кликов система отказалась от модального переключения между Claude и Gemini.
- **Inline Start:** Кнопки запуска обоих движков (`claude | gemini`) всегда видны в `InfoPanel`.
- **Context Preservation:** Выпадающее меню с дополнительным промптом (через Chevron) сохранено только для Claude, так как Gemini CLI обычно запускается в интерактивном режиме без предварительных инструкций.

## Командные кнопки (Status Styles)
Все интерактивные элементы управления командами (запуск, продолжение, экспорт) используют унифицированный стиль:
- **Style:** Тег `code`, фон `#1a1a1a` (глубокий черный), шрифт `text-xs`.
- **Hover:** Эффект `underline`.
- **Resume Button:** Кнопка «Продолжить» подстраивается под тип активной сессии (оранжевый текст для Claude, синий для Gemini), обеспечивая визуальную преемственность без перегрузки интерфейса цветами.

Для мгновенной визуальной ориентации пользователя в системе используется цветовое кодирование в зависимости от активного AI-движка:

- **Контекстные подписи:** Субтитры меняются на "Claude JSONL → clipboard" или "Gemini JSON → clipboard", подтверждая формат источника.

## 16. Multi-select Persistence (data-keep-selection)

### Проблема
При выборе нескольких вкладок (Shift+Click) любой клик по интерфейсу вне TabBar (например, в поле ввода промпта в ActionsPanel) приводил к сбросу выделения. Это делало невозможным массовое обновление документации с кастомным промптом.

### Решение: Exclusion Attribute
В `TabBar.tsx` (обработчик `mousedown` на `document`) добавлена проверка атрибута `data-keep-selection`.
- На контейнеры `ActionsPanel` добавлен этот атрибут.
- Если `e.target` или любой его родитель имеет этот атрибут — сброс выделения (`clearSelection`) **игнорируется**.
Это позволяет пользователю выделять вкладки, а затем свободно взаимодействовать с инструментами в боковой панели, не теряя контекст выбора.

## 18. Browser Tab Navigation (Contextual UX)

### Проблема: Разрыв контекста при клике по ссылке
В обычных терминалах Cmd+клик открывает ссылку в системном браузере ОС. Однако внутри «Браузерной вкладки» (BrowserTab) ожидаемое поведение иное: если пользователь запустил dev-сервер в этой же вкладке, он хочет видеть результат рядом. Открытие внешней ссылки разрушало этот контекст.

### Решение: Перехват и внутренняя навигация
Реализован проброс callback-а `onLinkClick` в компонент Terminal.
- **Motivation:** Мы разделяем «внешние» ссылки (открываются в ОС) и «контекстные» (в браузере, открываются внутри). Это превращает вкладку в полноценную IDE-подобную среду для веб-разработки.
- **Invisible Intent (Refresh):** Кнопка Refresh в браузере принудительно вызывает `setBrowserActiveView(tabId, 'browser')`. Это решение основано на паттерне «ожидаемого результата»: после ручного обновления страницы пользователь в 100% случаев хочет видеть UI, а не терминал, где он только что перезапустил сервер.

## Native Virtualization vs JS Libraries (The 200-Item Rule)

### Проблема: Прыжки скролла (Jitter) при динамической высоте
Использование библиотек виртуализации (Virtuoso, virtua) для списков с экстремальным разбросом высот (от 30px до 1000px, типично для Markdown-сообщений) приводит к рывкам при скролле вверх. Библиотеки "угадывают" высоту неотрендеренных элементов через `defaultItemHeight`, и когда реальный элемент оказывается больше/меньше — `scrollHeight` контейнера меняется, вызывая скачок позиции.

### Решение: Отказ от JS-библиотек в пользу Native CSS
Для списков до ~1000 элементов (типичная сессия Claude/Gemini) нативный скролл браузера работает стабильнее благодаря встроенному механизму **Scroll Anchoring**.

1.  **Нативный контейнер:** Простой `div` с `overflow-y: auto`.
2.  **`content-visibility: auto`**: Браузер сам пропускает layout и paint для элементов вне вьюпорта (аналог виртуализации на уровне движка).
3.  **`contain-intrinsic-size: auto 100px`**: Ключевой параметр. `auto` заставляет браузер запоминать реальную высоту элемента после первого рендера. При повторном скролле или изменении данных браузер использует кэшированную высоту, что полностью устраняет jitter.

### Преимущества
- **Стабильность:** Chromium компенсирует сдвиги на уровне композитора, что всегда плавнее JS-решений.
- **Производительность:** Нет накладных расходов на расчеты видимости в JS-потоке.
- **Вес:** Бандл уменьшается (минус ~20KB на библиотеку).
- **Интерактивность:** Раскрытие/скрытие блоков (аккордеонов) внутри списка не ломает скролл. Благодаря `contain-intrinsic-size: auto`, браузер мгновенно обновляет кэшированную высоту элемента сразу после его раскрытия пользователем.

## Interrupted Overlay Persistence (overlayDismissed lifecycle)

## Проблема
Приложение автоматически помечает активные сессии как `wasInterrupted = true` перед закрытием (`beforeunload`), чтобы предложить восстановление при следующем запуске.
Однако, если пользователь закрывал оверлей вручную (Escape или клик) и обновлял страницу, оверлей **появлялся снова**. Это происходило потому, что `beforeunload` срабатывал повторно и видел, что сессия всё ещё привязана к табу.

## Решение: Флаг `overlayDismissed`
В систему введено состояние "Осознанного закрытия":

1. **В Tab (SQLite):** Добавлена колонка `overlay_dismissed` (0/1).
2. **В UI:** При закрытии оверлея вызывается `dismissInterruptedSession()`, который ставит `overlayDismissed = true`.
3. **При закрытии:** Метод `markAllSessionsInterrupted()` теперь проверяет `if (hasSession && !overlayDismissed)`.
4. **Сброс флагов:** Флаги `wasInterrupted` и `overlayDismissed` сбрасываются в следующих случаях:
   - При нажатии кнопки "Продолжить" (Continue)
   - При полной очистке сессии таба
   - **При запуске НОВОЙ AI сессии** (`setClaudeSessionId`/`setGeminiSessionId`) — это гарантирует что overlay покажется снова если новая сессия будет прервана

## Жизненный цикл флагов

```
┌─────────────────────────────────────────────────────────────────┐
│  Запуск Claude/Gemini                                           │
│    setClaudeSessionId(tabId, newId)                             │
│    → wasInterrupted = false                                     │
│    → overlayDismissed = false   ← СБРОС при новой сессии        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Закрытие приложения (beforeunload)                             │
│    markAllSessionsInterrupted()                                 │
│    if (hasSession && !overlayDismissed)                         │
│      → wasInterrupted = true                                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Следующий запуск                                               │
│    if (wasInterrupted && claudeSessionId)                       │
│      → Показать Overlay "Восстановить сессию?"                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌───────────────────────────┬─────────────────────────────────────┐
│  Пользователь нажал       │  Пользователь закрыл Overlay        │
│  "Continue"               │                                     │
│    → claude --resume ID   │    → overlayDismissed = true        │
│    → wasInterrupted=false │    → Overlay не появится снова      │
│                           │      (пока не запустит НОВУЮ сессию)│
└───────────────────────────┴─────────────────────────────────────┘
```

## Результат
Пользователь видит оверлей только один раз за сессию. Если он его закрыл — выбор сохраняется. Но при запуске **новой** AI сессии флаги сбрасываются и цикл начинается заново.

## Manual Bounds Check for Hover states

## Проблема
В компоненте `BookmarkCard` при открытии выпадающего меню (троеточие) создается «бэкдроп» (`fixed inset-0`), который перекрывает всю область экрана. Когда пользователь кликает по бэкдропу, чтобы закрыть меню, браузер не генерирует событие `onMouseLeave` для карточки, так как мышь технически перешла на слой выше, но не «выехала» за границы элемента. В результате после закрытия меню карточка ошибочно оставалась в состоянии «Hover» (зеленый оверлей).

## Решение: Ручная проверка границ (Manual Bounds Check)

Вместо того чтобы полагаться на стандартные события браузера, была внедрена логика ручной проверки координат в момент клика.

### Алгоритм:
1. К компоненту привязывается `ref` (`cardRef`).
2. При клике на бэкдроп вызывается функция `handleBackdropClick(e)`.
3. Функция получает координаты клика: `clientX`, `clientY`.
4. Получает границы карточки через `cardRef.current.getBoundingClientRect()`.
5. Сравнивает координаты. Если клик произошел **вне** прямоугольника карточки, состояние `isHovered` принудительно сбрасывается в `false`.

```typescript
const isOutside =
  e.clientX < rect.left ||
  e.clientX > rect.right ||
  e.clientY < rect.top ||
  e.clientY > rect.bottom;

if (isOutside) {
  setIsHovered(false);
}
```

## Результат
Интерфейс ведет себя предсказуемо: если пользователь закрыл меню, кликнув в пустую область экрана, карточка мгновенно теряет фокус. Если кликнул внутри границ — фокус сохраняется.

## Global Terminal Selection Sync
**Файл-источник:** Сессия 2026-01-21

### Проблема
Компоненты (например, GeminiPanel) не могли получить выделенный в терминале текст без прямого доступа к инстансу `xterm.js`. Кнопки поиска не знали, когда текст выделен, и не могли быть заблокированы (disabled).

### Решение
Внедрен глобальный стейт `terminalSelection` в `useUIStore`.
1.  **Terminal.tsx:** Слушает событие `onSelectionChange` и обновляет глобальный стейт.
2.  **Context Menu:** Принудительно вызывает `getSelection()` перед открытием меню.
3.  **UI:** Кнопки поиска используют `terminalSelection` для управления состоянием `disabled` и отображения счетчика символов.
См. также: `knowledge/fact-fact-terminal-core.md`.

## UI Responsiveness / React 19 startTransition
**Файл-источник:** Сессия 2026-01-21 (Performance Optimization)

### Проблема
При переключении между "тяжелыми" проектами или открытии новых вкладок интерфейс замирал, не реагируя на клики, пока новый терминал не отрисуется полностью.

### Решение
Использование API `startTransition` в React 19 для всех операций смены контекста (`switchTab`, `showWorkspace`).
```javascript
import { startTransition } from 'react';

const switchTab = (tabId) => {
  startTransition(() => {
    set({ activeTabId: tabId });
  });
};
```

### Результат
React помечает обновление как "низкоприоритетное". Это позволяет браузеру продолжать обработку анимаций и кликов (например, подсветку таба при наведении), пока тяжелая работа по рендерингу сетки терминала происходит в фоновом потоке.

## 17. Double Debounce Strategy for Timeline (section 17)

### Проблема
Timeline должен отслеживать видимость сообщений во вьюпорте и их наличие в полном буфере. Прямая проверка на каждый скролл или вывод данных (onWriteParsed) вызывает сотни тяжелых строковых операций в секунду.

### Решение: Трёхуровневая задержка
1.  **Viewport Highlight (100ms):** Легкая операция. Проверяет только те строки, что видны на экране. Срабатывает быстро, обеспечивая отзывчивость "черточки".
2.  **Reachability Check (500ms):** Тяжелая операция. Сканирует **весь** буфер терминала (включая scrollback). Выполняется реже, чтобы не нагружать процессор при активном выводе текста.
3.  **Data Sync (2000ms):** Самая тяжелая. Загрузка JSONL с диска и перестроение дерева Backtrace.
Такое разделение позволяет Timeline выглядеть "живым" без ущерба для производительности терминала.

## 19. Floating Panels Alignment (Precise Positioning)

### Проблема: Хрупкость JS-расчетов при динамическом контенте
Изначально панели "Параметры" и "Настройки" позиционировались через расчет `left: blockRect.left - panelWidth`. Это приводило к ошибкам:
1. Если контент панели менялся (например, длинный текст перевода "Редактирование"), хардкод `panelWidth` переставал соответствовать реальности, и панель либо наезжала на блок, либо улетала слишком далеко.
2. При использовании Portals возникали микро-задержки между получением координат и отрисовкой, что вызывало визуальное "дерганье".

### Решение: Anchored CSS Transforms
Вместо расчета координат для левого края панели, мы устанавливаем её левый край в точку начала блока и сдвигаем саму панель на её полную ширину через CSS.

1.  **Точка привязки:** `left: blockRect.left - 3px` (небольшой отступ).
2.  **Трансформация:** `transform: translate(-100%, -50%)`.
    - `-100%` (X) — сдвигает панель влево ровно на её текущую ширину. Правый край панели всегда оказывается точно в точке привязки.
    - `-50%` (Y) — идеально центрирует панель по вертикали относительно центральной точки блока.
3.  **Логические мостики:** Отказ от "физических" прозрачных `div` (bridge divs) в пользу логической проверки в `onMouseLeave`. Если мышь движется в сторону портала (влево), мы не закрываем меню. Это упрощает DOM и делает позиционирование независимым от размеров мостика.

### Результат
Панели всегда приклеены "встык" к блоку, независимо от их содержимого и текущего языка интерфейса.

## Compact paths + instant tooltips + live path validation + prepare-before-show (Polish section)

## 1. Компактные пути (Dashboard)
Чтобы не забивать интерфейс длинными строками типа `/Users/fedor/Projects/Web/my-app`, реализована функция сокращения пути.
- **Логика:** В карточке проекта отображаются только последние две директории.
- **Реализация:** `parts.slice(-2).join('/')`.

## 2. Мгновенные тултипы (Project Paths)
Поскольку сокращенный путь может быть неоднозначным, внедрен механизм мгновенного показа полного пути.
- **Behavior:** При наведении на сокращенный путь в `ProjectCard` полный адрес всплывает без задержки.
- **Implementation:** Использование стандартного атрибута `title` в сочетании с кастомным `Tooltip` компонентом для консистентности.

## 3. Живая валидация путей (Edit Modal)
Для предотвращения ошибок при ручном вводе или переименовании путей внедрена фоновая проверка.
- **Mechanism:** IPC-хендлер `project:validate-path` использует `fs.existsSync`.
- **UI:** Иконка ✅ появляется, если путь валиден, и ⚠️ — если папка не найдена.

## 4. Фоновая подготовка (Prepare-before-show)
Для устранения «эффекта белого экрана» при создании или переключении проекта:
1. Система сначала генерирует ID и создает записи в БД.
2. Инициализирует PTY-сессии.
3. Дожидается готовности рендерера терминала.
4. Только после этого убирает `RestoreLoader` и показывает проект.

## Immediate Injection (AI session flicker fix)

## Problem
При использовании Rollback или создании форка, правая панель (InfoPanel) на короткое время (300-500мс) показывала состояние "Нет активной сессии", хотя сессия восстанавливалась.

## Symptoms
- Визуальный "прыжок" интерфейса.
- Скрытие кнопок управления сессией сразу после нажатия Rollback.

## Cause: Race Condition
1. `closeTab()` обнуляет `geminiSessionId` в сторе.
2. `createTab()` создает пустой таб.
3. InfoPanel через поллинг видит `null` и рендерит пустой экран.
4. `executePendingAction` в `Terminal.tsx` срабатывает только ПОСЛЕ готовности PTY.
5. Только тогда вызывается `setGeminiSessionId()`, и InfoPanel "просыпается".

## Solution: Immediate Injection
Изменен метод создания вкладок для AI-команд. Теперь `geminiSessionId` / `claudeSessionId` передается в `options` метода `createTab` или `createTabAfterCurrent`.
- Таб создается **уже** с ID сессии.
- InfoPanel видит ID мгновенно, не дожидаясь готовности PTY.

### Применение паттерна
1. **Fork/Rollback:** При создании форка или отката ID сессии передаётся в `createTab()` напрямую.
2. **History Restore:** При восстановлении вкладки из History (ProjectHome) `claudeSessionId` / `geminiSessionId` берутся из SQLite и передаются в `createTab()`. Sniper Watcher **не нужен** — ID уже известен. См. `knowledge/fact-project-home.md`.

## Результат
Бесшовный переход между вкладками и мгновенное отображение статуса сессии после отката или восстановления из истории.

## Interactive Hover Zones / Invisible Bridge Pattern

## Проблема
При использовании **React Portals** (например, для тултипов или превью сообщений) всплывающий элемент рендерится в конце `<body>`, вне иерархии DOM триггера. Это делает невозможным использование `relatedTarget` в `onMouseLeave`, так как для браузера это абсолютно разные ветки.

**Следствие:** При попытке переместить курсор от "точки" (триггера) к "окну" (превью) оно мгновенно закрывается, так как между ними есть физический разрыв.

## Решение: Invisible Bridge
Для создания плавного перехода используется стратегия "Невидимого мостика":

1.  **Логический мостик (Directional Closing):** Вместо таймеров или дополнительных прозрачных слоёв используется проверка направления движения мыши в `onMouseLeave` триггера.
    - Если мышь ушла в сторону портала — игнорируем закрытие.
    - Если в любую другую сторону — закрываем мгновенно.
2.  **No Scale Policy:** Триггеры, открывающие сложные интерактивные зоны, не должны использовать `transform: scale` при hover. Изменение размеров элемента "на лету" меняет его `getBoundingClientRect`, что создаёт микро-разрывы между триггером и порталом.

### Преимущества
- **Чистый DOM:** Нет лишних прозрачных `div` слоёв, которые могут мешать кликам по другим элементам.
- **Точность:** Логика срабатывает мгновенно и не зависит от скорости движения мыши (в отличие от дебаунса).

## Примеры реализации
- `docs/knowledge/fact-timeline.md`: Превью сообщений.
- `docs/knowledge/fix-gemini-response-queue.md`: Очередь ответов (SubAgentBar).
- `src/renderer/components/Workspace/panels/ActionsPanel.tsx`: Меню настроек копирования (📋), API Settings (⚙).

## Когда применять
Всегда, когда пользователю нужно взаимодействовать с контентом всплывающего окна, которое открывается по hover.

## Cmd+Hover Settings Panel (Конкретный шаблон)

### Концепция
Плавающая панель настроек, которая появляется при зажатом Cmd и наведении на триггер (иконку). Панель позиционируется слева от триггера через Portal и остаётся открытой при перемещении курсора внутрь неё — даже после отпускания Cmd.

### Почему не таймеры и не pinned-стейт
Пробовали `setTimeout` + `isPinned` + `gearCloseTimer` — лишнее усложнение. Directional Closing + реактивное вычисление видимости решает все кейсы без таймеров.

### Реактивная видимость (ключевое отличие от обычного hover)
Обычный hover (`onMouseEnter` → show) не работает, когда курсор уже на месте и пользователь нажимает/отпускает Cmd. `onMouseEnter` не стреляет повторно. Решение — **разделить hover-стейт и видимость**:

```tsx
const [gearHovered, setGearHovered] = useState(false);       // мышь над триггером
const [gearPanelHovered, setGearPanelHovered] = useState(false); // мышь внутри панели

// Видимость — derived state, не отдельный useState
const showPanel = (isCmdPressed && gearHovered) || gearPanelHovered;
```

`isCmdPressed` — из хука `useCmdKey()`. При нажатии/отпускании Cmd React делает ре-рендер → `showPanel` пересчитывается → панель появляется/исчезает без участия mouse events.

### Триггер (иконка ⚙)
```tsx
onMouseEnter={() => setGearHovered(true)}
onMouseLeave={(e) => {
  // Directional: закрыть только если мышь ушла НЕ в сторону панели
  const rect = e.currentTarget.getBoundingClientRect();
  if (e.clientX >= rect.left) {  // панель слева → пропускаем уход влево
    setGearHovered(false);
  }
}}
```

### Portal-панель
```tsx
onMouseEnter={() => setGearPanelHovered(true)}
onMouseLeave={() => { setGearPanelHovered(false); setGearHovered(false); }}
```
При входе в панель — pin (через `gearPanelHovered`). При выходе — полный сброс.

### Позиционирование
```tsx
style={{
  position: 'fixed',
  left: rect.left - 3,
  top: rect.top + rect.height / 2,
  transform: 'translate(-100%, -50%)',  // панель слева, вертикально по центру
  zIndex: 10000,
}}
```
Паттерн Anchored CSS Transforms — правый край панели всегда встык к триггеру, независимо от ширины контента.

### Матрица состояний

| Действие | `isCmdPressed && gearHovered` | `gearPanelHovered` | Панель |
|---|---|---|---|
| Курсор на ⚙, нажал Cmd | true | false | видна |
| Курсор на ⚙, отпустил Cmd | false | false | скрыта |
| Переместил курсор в панель | — | true | видна |
| Отпустил Cmd в панели | false | true | видна |
| Ушёл из панели | — | false | скрыта |

### Применение
- `ActionsPanel.tsx`: ⚙ API Settings (модель/thinking для Update Docs)
- `ActionsPanel.tsx`: 📋 Copy Settings (Чтение, Редактирование, От fork) — использует упрощённый вариант без Cmd (обычный hover + directional)
- `TabBar.tsx`, `ProjectHome.tsx`, `Dashboard.tsx`: `useCmdHoverPopover` хук — обобщённая версия этого паттерна для списков элементов

## State-driven Async Refresh (API Sync)

### Проблема
Когда Main-процесс изменяет данные в БД (например, записывает результат `adoptClaudeAgent`), Renderer-процесс не узнает об этом мгновенно. Использование чистого поллинга (polling) создает либо лишнюю нагрузку, либо визуальную задержку.

### Решение: Event-driven Trigger
UI компоненты, отображающие системные логи или сессии, используют паттерн "Слушатель + Обновитель":
1. **Trigger:** Main-процесс отправляет IPC-событие (например, `mcp:agent-adopted`) сразу после успешной записи в БД.
2. **Listener:** Renderer слушает это событие и вызывает функцию загрузки данных (`loadApiCalls`).
3. **Transition Logic (500ms Delay):** В обработчик события добавляется задержка 500мс перед вызовом `loadApiCalls`.
    - **Motivation:** Это компенсирует время, необходимое SQLite для фиксации транзакции (commit) на диске, предотвращая ситуацию, когда UI пытается прочитать файл, который еще не полностью обновлен ОС.

### GPU-Accelerated Loaders
Во время ожидания данных в список вставляется "строка-призрак" с анимированным лоадером. Лоадер обязательно использует `will-change: transform`, чтобы не блокироваться тяжелым рендером соседних элементов. См. раздел "Compositor-Thread Animations".

## Home-Dashboard Toggle (Navigation Context)

### Концепция
Кнопка 🏠 (Home) работает как **интеллектуальный переключатель контекста**, а не простая навигация. Это минимизирует клики при частом переходе между Dashboard и активным проектом.

### Поведение
- **На Dashboard:** Повторный клик по 🏠 возвращает пользователя в контекст **последнего активного проекта**, восстанавливая прежнее состояние (активный таб, позиция в истории, развёрнутые боковые панели).
- **Вне Dashboard:** Клик по 🏠 переводит в Dashboard.

### Реализация
1. При входе на Dashboard система сохраняет ID последнего активного проекта в состояние `lastActiveProjectIdBeforeDashboard`.
2. При клике на 🏠 во время просмотра Dashboard система вызывает `switchToProject(lastActiveProjectIdBeforeDashboard)`.

### Мотивация
Паттерн "Back to context" снижает навигационную нагрузку на пользователя. Вместо двух кликов (Dashboard → Project Name) требуется один. Особенно полезно при создании новых проектов и возврате в Dashboard для проверки списка.

## Silent Automation Cleanup (skipProcessCheck Flag)

### Концепция
Программные инструменты (MCP tools, Update API pipeline, Sub-Agent Orchestration) нередко закрывают вкладки с активными процессами программно. Стандартная логика закрытия вкладок вызывает диалоговое окно "У вас запущен процесс..." — это ломает flow пользователя и замораживает автоматизацию.

### Решение: skipProcessCheck Flag
При программном закрытии вкладки передаётся флаг `skipProcessCheck: true` в `closeTab()`.

```typescript
// Программное закрытие (например, после отмены Update Docs)
closeTab(tabId, { skipProcessCheck: true });

// Обычное закрытие пользователем (показать диалог, если процесс активен)
closeTab(tabId);
```

### Применение
- **MCP `close_sub_agent` tool:** Закрывает таб с suб-агентом без алертов.
- **Update Docs отмена:** При прерывании процесса Gemini-таб закрывается бесшовно (`gemini:close-tab` с `skipProcessCheck`).
- **Orchestration cleanup:** При восстановлении сессии система закрывает вспомогательные табы без спама диалогов.

## Scroll-to-Bottom Arrow Pattern

### Концепция
Для длинных вертикальных списков (Timeline, History Panel), где пользователь может потерять контекст последних сообщений при скролле вверх, используется унифицированный паттерн кнопки возврата к актуальным данным.

### Реализация
- **Визуал:** Компактная круглая кнопка с иконкой `ChevronDown`.
- **Порог появления:** Кнопка становится видимой только когда расстояние от текущей позиции скролла до нижней границы контейнера превышает **150px**.
- **Скрытие:** Кнопка мгновенно скрывается (`display: none`), как только пользователь достигает "зоны дна" (менее 5-10px до конца).
- **Скролл:** Используется плавный переход (`behavior: 'smooth'`) до максимально возможного значения `scrollTop`.

### Преимущества
- **Когнитивный комфорт:** Пользователь всегда знает, есть ли "ниже" новые данные, не проверяя скроллбар.
- **Предотвращение лишних кликов:** Одно нажатие возвращает к самому свежему выводу AI.
- **Производительность:** Видимость кнопки управляется через прямую манипуляцию `style.display` в обработчике скролла, минуя тяжелые циклы ре-рендеринга React.

## Mirror Editing (Double-Column Layout)

### Концепция
Паттерн взаимного исключения редакторов в двухколонной раскладке настроек (Settings → AI). Вместо открытия модальных окон поверх списка, форма редактирования заменяет собой соседнюю колонку.

### Реализация
- **Левая колонка (AI промпты):** При клике на карточку редактор открывается в правой колонке, замещая список текстовых вставок.
- **Правая колонка (Вставки):** При клике на карточку редактор открывается в левой колонке, замещая список AI промптов.

### Преимущества
- **Контекст:** Пользователь всегда видит список элементов той категории, которую он НЕ редактирует в данный момент.
- **Стабильность:** Отсутствие Layout Shift всего модального окна — ширина колонок остается фиксированной (50/50), меняется только их содержимое.
- **Фокус:** Единственный активный редактор на экране предотвращает когнитивную перегрузку и ошибки при сохранении.
