# Fix: MCP Completion Reliability (The turn_duration Guard)

## Проблема: Ложное завершение (False Idle)
При делегации задач суб-агентам Claude Code (инструмент `Task`), приложение часто детектировало завершение работы слишком рано, впрыскивая в Gemini промежуточные сообщения вместо финальных ответов.

### Причина: Микро-паузы Ink TUI
Claude Code использует Ink (React TUI), который перестает отрисовывать спиннер (`✶`) в короткие промежутки времени:
1. Клод выводит текстовый блок → спиннер исчезает.
2. Клод делает API-вызов для запуска суб-агента (Task) → возникает пауза > 500мс.
3. Наш `busy-state` детектор видит отсутствие спиннера и тишину в потоке → переключается в `IDLE`.
4. Система считает задачу выполненной, хотя Клод только начал работать над инструментами.

## Решение: Детерминированный сигнал завершения (JSONL Guard)
Вместо того чтобы полагаться только на визуальный IDLE, система теперь использует `turn_duration` как единственный 100% надежный признак окончания хода.

### 1. turn_duration Сигнал
Claude Code записывает объект `system:turn_duration` в JSONL-файл сессии **только** после того, как весь ход (включая все вложенные вызовы инструментов) полностью завершен.

### 2. Алгоритм Deferred Re-check
При переходе спиннера в состояние `IDLE` запускается процедура верификации:
- **Check JSONL:** Система читает хвост (последние 10 записей) JSONL-файла.
- **Pass:** Если найден `turn_duration` — задача считается выполненной, результат извлекается и доставляется.
- **Defer:** Если `turn_duration` отсутствует, но есть признаки активности (`progress`, `tool_use` или `queue-operation`), статус `running` сохраняется.
- **Re-check:** Запускается таймер на **3 секунды** для повторной проверки. Всего допускается до 40 попыток (2 минуты ожидания тишины).
- **Interrupt:** Если во время ожидания спиннер снова стал `BUSY`, отложенная проверка мгновенно отменяется.

### 3. Поддержка очередей (queue-operation)
Если Gemini присылает новые сообщения через `continue_claude` пока Клод занят, они попадают в очередь. Появление записей `queue-operation` в логе теперь также считается признаком "продолжающейся работы", блокирующим ложное завершение.

## Invisible Intent
Не пытайтесь "оптимизировать" логику, убирая 3-секундную задержку. Она необходима, так как запись в JSONL на диске часто отстает от визуального рендеринга в терминале на сотни миллисекунд. Требование явного `turn_duration` — это единственный способ гарантировать целостность доставленного ответа.
