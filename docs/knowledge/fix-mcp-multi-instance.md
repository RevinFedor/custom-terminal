# Fix: Multi-instance MCP Isolation (PID-based Ports)

## Проблема
Изначально приложение использовало единый файл `~/.noted-terminal/mcp-port` для передачи номера порта HTTP-сервера MCP-клиенту (`mcp-server.mjs`). Это создавало критические гонки (race conditions) при параллельном запуске нескольких экземпляров приложения (например, dev-версия и prod-версия, или два окна разработки).

### Симптомы
- Gemini в одном окне делегирует задачу, но она "приземляется" в терминале другого окна.
- Ошибка "MCP port file not found", если один экземпляр закрылся и удалил общий файл, пока второй ещё работает.
- Непредсказуемое поведение кнопок статуса и вставки результатов.

## Root Cause: Race Condition
1. **`startMcpHttpServer()`** → записывал порт в фиксированный путь. Последний запустившийся инстанс "побеждал", перезаписывая порт предшественника.
2. **`stopMcpHttpServer()`** → вызывал `unlinkSync` на общем файле при выходе. Это лишало связи все остальные работающие инстансы.

## Решение: PID-based Port Files
Система переведена на использование именных файлов портов, привязанных к PID главного процесса Electron.

1. **Запись:** Каждый инстанс пишет в `~/.noted-terminal/mcp-port-<process.pid>`.
2. **Очистка (Cleanup):** При старте приложение сканирует папку портов и удаляет файлы, чьи PID больше не существуют в системе (проверка через `process.kill(pid, 0)`).
3. **Удаление:** При нормальном выходе удаляется только файл, соответствующий текущему PID.

## Алгоритм обнаружения (Process Tree Walking)
Поскольку `mcp-server.mjs` запускается самим Gemini CLI (который запущен в шелле, который запущен в `node-pty`), он не знает PID "своего" Electron напрямую. 

Для поиска правильного порта реализован **Process Tree Walking**:
1. **Цепочка:** `mcp-server (node)` → `Gemini CLI` → `zsh/bash` → `Electron (node-pty)`.
2. **Strategy 1 (Exact Match):** Сервер берет свой PPID (Gemini) и поднимается вверх по дереву процессов (до 4 уровней). Для каждого родительского PID проверяется наличие файла `mcp-port-<PID>`. Это гарантирует точное попадание в тот инстанс Electron, который породил текущую сессию Gemini.
3. **Strategy 2 (Liveness Fallback):** Если дерево процессов недоступно, сканируются все `mcp-port-*` файлы. Выбирается самый свежий по `mtime`, чей процесс еще жив.
4. **Strategy 3 (Backward Compat):** Попытка прочитать старый `mcp-port` (для плавного перехода).

## Invisible Intent: Почему не `execSync` в Main
Согласно правилам `CLAUDE.md`, Main-процесс не должен блокироваться на системных вызовах. Поэтому тяжелая работа по обходу дерева процессов (`ps -p PID -o ppid=`) вынесена в `mcp-server.mjs`, который является независимым процессом Node.js.

## Связанные файлы
- `src/main/main.js` — реализация записи и очистки портов.
- `src/main/mcp-server.mjs` — реализация алгоритма поиска порта.
- [`knowledge/fact-mcp-delegation.md`](fact-mcp-delegation.md) — общая архитектура делегации.
