# Feature: Gemini → Claude MCP Delegation

## Intro
Система делегации позволяет Gemini CLI передавать сложные задачи Claude Code, запуская его в полноценном, видимом пользователю терминальном табе. Это превращает Gemini в "оркестратора", который может наблюдать за работой sub-агента.

## Архитектура: Stateful Delegation (Сессионная делегация)
В отличие от ранних версий, текущая система MCP-делегации поддерживает долгоживущие сессии суб-агентов. Приложение выступает как «интеллектуальный роутер», сохраняющий состояние между вызовами.

### Основные инструменты:
1. **`delegate_to_claude(prompt, name)`**: Создает новую вкладку и запускает Claude. Опциональный параметр `name` позволяет сразу задать заголовок вкладки. Возвращает `taskId`.
2. **`continue_claude(taskId, prompt, name)`**: Отправляет сообщение в существующую сессию. Параметр `name` позволяет переименовать вкладку «на лету». Если приложение было перезапущено, автоматически выполняет `claude --resume` в том же табе.
3. **`list_sub_agents()`**: Возвращает Gemini список всех его суб-агентов (включая тех, что были созданы до перезапуска приложения) с их именами и статусами.
4. **`read_claude_history(taskId, detail, from_beginning)`**: Позволяет Gemini получить доступ к логам Claude. По умолчанию возвращает только новые сообщения с момента последней доставки/чтения (инкрементально). Параметр `from_beginning: true` позволяет получить всю историю целиком.

### Project Association Guard (Многопроектная изоляция)
При создании суб-агента система гарантирует его привязку к тому же проекту, в котором запущен родительский Gemini.
- **Проблема:** Использование `activeProjectId` (текущий вид в UI) приводило к тому, что при фоновой работе оркестратора вкладки Claude «улетали» в тот проект, который пользователь просматривал в данный момент.
- **Решение:** Система итерирует по всем открытым проектам (`openProjects`) для точного поиска владельца `geminiTabId`. Это обеспечивает корректную изоляцию контекстов при параллельной работе в нескольких окнах.

### Visibility Defense (Кросс-проектная видимость)
Для предотвращения «исчезновения» агентов при ошибках ассоциации или перемещении вкладок, панель `SubAgentBar` использует алгоритм сквозного поиска.
- **Механика:** Система ищет дочерние вкладки (`parentTabId`) во всех открытых проектах приложения, а не только в текущем воркспейсе.
- **Motivation:** Связь «родитель-ребенок» является первичной по отношению к принадлежности к проекту. Это гарантирует, что пользователь всегда увидит своих суб-агентов над терминалом Gemini, даже если они технически были сохранены или перемещены в другой проект.

### Механика доставки ответа:
Результат работы Claude **никогда не возвращается** напрямую в Tool Result.
1. MCP-сервер мгновенно возвращает `taskId`, подтверждая прием задачи.
2. Main-процесс отслеживает завершение работы Claude через PTY-поток (Spinner Detection).
3. При переходе Claude в состояние IDLE, результат извлекается из логов.
4. **Response Queue:** Перед инъекцией система проверяет три условия:
   - `geminiSpinnerBusy` — Gemini обрабатывает запрос (спиннер активен).
   - `geminiInputHasText` — пользователь набирает текст в input-поле Gemini.
     - **Invisible Intent (Character Counting):** Система отслеживает нажатия клавиш в `terminal:input`. В отличие от Phase 1, теперь поддерживается детекция удаления текста (Backspace/Ctrl+U). Это позволяет автоматически вставлять ответы, как только пользователь очистил строку.
   - `geminiResponseQueue` — наличие уже ожидающих ответов от других агентов.
   - Если любое условие истинно → ответ ставится в очередь. Очередь обрабатывается последовательно при переходе Gemini в IDLE и полной очистке input (счётчик = 0).
5. Ответ впрыскивается в Gemini PTY через `safePasteAndSubmit` в тегах `[Claude Sub-Agent Response]`. См. детали в [`fix-gemini-response-queue.md`](fix-gemini-response-queue.md).

### Поиск суб-агента (3-Level Fallback):
Для инструмента `continue_claude` система ищет таб по следующим уровням:
1. **Memory:** Активная запись в `mcpTasks` (UUID задачи).
2. **Parent Link:** Маппинг в `subAgentParentTab` (связь живых PTY).
3. **Persistence (SQLite):** Прямой запрос в БД `SELECT tab_id FROM tabs WHERE parent_tab_id = ?`. Это позволяет восстанавливать сессии даже после полной очистки оперативной памяти.

### Психология промпта (Tool Descriptions)
Для предотвращения бесконечных циклов в Gemini ("Potential loop detected") описания инструментов в MCP-сервере сформулированы максимально жестко:
- **Asynchronous:** В описании инструментов явно указано, что результат появится в терминале сам собой на "мета-уровне". Gemini **НЕ ДОЛЖЕН** поллить статус задачи.
- **Diagnostic only:** Инструмент `get_task_status` помечен как "только для диагностики", чтобы агент не использовал его в основном цикле ожидания.

### Handshake: Sequential Command Splitting
При делегации промпта, содержащего слэш-команды (например, `/model haiku\nTask`), система использует механизм **Sequential Dispatch**:
1. **Splitting:** Текст разбивается на команды и основное тело.
2. **Sequential Send:** Команды отправляются по одной через `safePasteAndSubmit`.
3. **Ready Wait:** После каждой команды система ждет возврата промпта Claude (`⏵` или `>`) в PTY-потоке. 
   - **Motivation:** Без ожидания основное тело задачи могло "улететь" в Claude, пока тот еще переключает модель, что приводило к ошибкам "Model not found".
4. **Body Injection:** Основной промпт вставляется только после подтверждения готовности.

## Механизм обнаружения (PID-based Discovery)
Main-процесс запускает HTTP-сервер на случайном свободном порту (`port 0`).
- **Endpoint Discovery:** Актуальный порт записывается в `~/.noted-terminal/mcp-port-<process.pid>`. Использование PID в имени файла обеспечивает изоляцию при параллельном запуске нескольких экземпляров приложения.
- **MCP Server:** Процесс `mcp-server.mjs` при старте выполняет **Process Tree Walking** вверх от своего PPID (Gemini), чтобы найти файл порта, соответствующий "родному" инстансу Electron. См. [`fix-mcp-multi-instance.md`](fix-mcp-multi-instance.md).

## Reliability & Completion (JSONL Guard)
Для обеспечения 100% точности доставки ответов система использует двухуровневую проверку завершения:
1. **Spinner IDLE:** Триггерится при отсутствии Braille-спиннера в потоке PTY в течение 500мс.
2. **JSONL Guard:** Система проверяет лог сессии на наличие поля `system:turn_duration`. 
   - Если поле отсутствует — задача считается продолжающейся (даже если спиннера нет).
   - Применяется **Deferred Re-check** (3с интервал) для компенсации задержек записи на диск.
   - Подробнее: [`fix-mcp-completion-reliability.md`](fix-mcp-completion-reliability.md).

## Viewport UX: Seamless Observation
При делегации создается связь `parentTabId`. Это активирует специальное поведение интерфейса:
- **SubAgentBar:** Над терминалом появляется панель с чипами активных sub-агентов.
- **Queue Indicator:** Если ответы субагентов находятся в очереди (из-за занятости Gemini или ввода пользователя), в правой части `SubAgentBar` появляется фиолетовый индикатор «Queue: N» с пульсирующей точкой (соответствует цвету UUID в терминале).
    - **Actions:** Справа от бейджа доступна кнопка **«Send now»** (зеленая), которая принудительно очищает инпут и впрыскивает следующий ответ.
    - **Tooltip:** При наведении на бейдж показывается превью промптов всех ожидающих ответов. Используется **Invisible Bridge**, чтобы тултип не закрывался при движении к нему.
- **UUID Links in Terminal:** В выводе Gemini (терминал) UUID задач автоматически распознаются как ссылки. Клик по ID вызывает переход к соответствующей вкладке Claude-суб-агента через IPC `mcp:focus-task`. См. [`fix-tab-persistence.md`](fix-tab-persistence.md) (секция Closure Trap).
- **Viewport Switching:** Клик по чипу переключает видимость (`zIndex`) между терминалом Gemini и Claude. 
- **Active Context:** Фокус в `TabBar` остается на Gemini. Это позволяет пользователю "подглядывать" за работой Claude, не переключая основной контекст вкладки.

### Изоляция (Visibility)
Вкладки суб-агентов **жестко фильтруются** в основном `TabBar`.
- **Правило:** Таб с установленным `parentTabId` никогда не отображается в основном списке (ни в Main, ни в Utility зонах). См. [`fix-tab-persistence.md`](fix-tab-persistence.md) для деталей сохранения этой связи после рестарта.
- **Motivation:** Это предотвращает захламление интерфейса и четко разделяет "оркестратор" (Gemini) и его "рабочих воркеров" (Claude sub-agents).
- **Detach:** При отсоединении суб-агента (через контекстное меню чипа), `parentTabId` сбрасывается в `undefined`, после чего таб мгновенно появляется в общем списке `TabBar`.

## Behavior Specs
- **Status Sync:** Чипы sub-агентов в реальном времени отображают статус (`⟳` running, `✓` done, `✗` error). При перезапуске приложения статус восстанавливается как `done`.
- **Timeout Monitoring:** Система отслеживает время выполнения каждой задачи (30 минут). Время старта фиксируется в БД (`mcp_task_started_at`), и таймер обнуляется при каждом продолжении работы (`continue_claude`). См. [`fix-mcp-timeout-stability.md`](fix-mcp-timeout-stability.md).
- **Cleanup:** При закрытии Gemini-таба все связанные с ним sub-агент процессы получают сигнал `Ctrl+C` и отключаются. Таймеры при этом принудительно очищаются.
- **Sub-Agent Context Menu:** Правый клик по чипу в SubAgentBar открывает нативное меню:
    - **Task ID:** Копирование ID задачи (нужно для диалога с Gemini).
    - **Session ID:** Копирование ID сессии Claude (для Fork/Timeline).
    - **Detach:** Отсоединение таба и превращение его в самостоятельную вкладку.
