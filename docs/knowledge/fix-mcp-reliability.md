# Fix: MCP Delegation Reliability & Edge Cases

Этот файл описывает решения критических проблем, возникших при реализации Agent-to-Agent взаимодействия через реальные PTY-табы.

---

## 1. The Cooldown Deferral (Early Completion Race)

### Проблема
Claude CLI при запуске сразу выводит промпт `⏵`. MCP-детектор завершения (`Completion Detector`) видел этот символ и мгновенно закрывал задачу как "выполненную" еще до того, как Handshake успевал отправить реальный промпт пользователя.

### Решение: Отложенная детекция (Deferral)
Был внедрен 5-секундный защитный интервал (`cooldown`) после отправки промпта.
- **Ловушка:** Если Claude отвечал быстрее 5 секунд, детектор "пропускал" финальный промпт и задача зависала в статусе `running` навсегда.
- **Финальное решение:** Вместо игнорирования событий во время cooldown, система **откладывает** (defer) проверку. Если `⏵` замечен во время защиты, планируется отложенная проверка JSONL сразу после истечения таймера. Это гарантирует, что даже мгновенные ответы будут захвачены.

## 2. CWD Cross-Project Fix

### Проблема
Sub-agent табы создавались с невалидными путями (например, `__unset__...`), что приводило к краху Claude при старте.
**Причина:** Renderer-процесс пытался вычислить CWD, основываясь только на активном проекте. Если Gemini-таб находился в другом проекте (параллельные окна), поиск возвращал ошибку.

### Решение: Main-Process Source of Truth
Логика резолвинга CWD перенесена в Main-процесс. При запросе создания таба Main-процесс сам находит актуальный путь из `terminalProjects` Map и передает его в renderer через IPC. Renderer обязан использовать `data.cwd` из сообщения, если он предоставлен.

## 3. Terminal Spawn Retry

### Проблема
Иногда sub-agent таб создавался успешно, но попытка отправить в него команду завершалась ошибкой `Claude terminal not found`.

### Причина: IPC Race Condition
Renderer генерирует `tabId` и посылает `terminal:create`, а затем сразу сообщает Main-процессу о готовности через `mcp:sub-agent-tab-created`. Из-за асинхронности PTY-процесс мог еще не успеть зарегистрироваться в глобальном Map `terminals` в Main-процессе.

### Решение
В функцию делегации добавлен цикл повторных попыток (**Retry Loop**): 5 попыток с задержкой 500мс. Это дает системе время на завершение регистрации PTY до начала взаимодействия с ним.

## 5. Slash-Command Splitting (Handshake Fix)

### Проблема
Gemini CLI часто делегирует задачи в формате:
```
/model haiku
Проведи исследование этого проекта...
```
Система Handshake воспринимала это как один текстовый блок. Claude Code, получив это через Bracketed Paste, пытался найти модель с именем `"haiku\nПроведи исследование..."` и выбрасывал ошибку.

### Решение: `sendHandshakePrompt`
Внедрена логика парсинга промпта перед отправкой:
1. **Extraction:** Если первая строка начинается с `/`, она извлекается как отдельная slash-команда.
2. **Sequential Send:** Slash-команда отправляется первой через `safePasteAndSubmit`.
3. **Prompt Return Wait:** Система ждет возврата промпта Claude (`⏵` или `>`) через PTY-слушатель (с 10-секундным таймаутом).
4. **Body Injection:** Только после этого отправляется основной текст промпта.

Это гарантирует, что Claude успеет переключить модель до того, как получит саму задачу.

## 6. Status Propagation (Status Sync)

### Проблема
Индикаторы на чипах суб-агентов в `SubAgentBar` либо не мигали, либо мигали вечно (даже после завершения задачи).

### Причина
Рендерер не слушал события `mcp:task-status` от Main-процесса. Приложение знало, что задача завершена, но `claudeAgentStatus` в сторе оставался `undefined`.

### Решение
1. **IPC Bridge:** В `App.tsx` добавлен глобальный слушатель `mcp:task-status`.
2. **Mapping:** Статусы MCP (`running`, `completed`, `error`, `timeout`) мапятся на внутренние статусы агента (`running`, `done`, `error`).
3. **Store Update:** При получении статуса вызывается `setClaudeAgentStatus(tabId, status)`, что мгновенно обновляет визуальное состояние чипа (остановка пульсации, смена иконки на ✓ или ✗).
