## Как сейчас работает таймлайн

1. **Данные**: Каждые 2 секунды Timeline.tsx загружает записи из JSONL-файлов сессии Claude через IPC `claude:get-timeline`. Backtrace-алгоритм восстанавливает активную ветку диалога.

2. **Визуал**: Вертикальная полоска 24px справа от терминала. Каждая запись = кружочек (4px, `rgba(255,255,255,0.3)`). При hover — 8px, белый с glow.

3. **Навигация**: Клик по кружочку → `terminalRegistry.searchAndScroll(tabId, text)` — ищет первые 40 символов промпта в буфере xterm.js и скроллит туда.

4. **Маркеры**: Уже есть marker API в `terminalRegistry.ts` — `bindEntryBySearch()`, `getEntryMarkerLine()`, но **markers не используются для подсветки видимости**.

5. **Viewport**: `Terminal.tsx:328` уже синхронизирует viewport в registry при каждом скролле (`updateViewport`). Timeline может подписаться через `onViewportChange`.

## Что нужно для подсветки "видимого" кружочка

Инфраструктура **уже почти готова**:
- `updateViewport()` — отслеживает `{top, bottom, total}` буфера
- `onViewportChange()` — callback при скролле
- `getEntryMarkerLine()` — возвращает строку буфера для UUID
- `bindEntryBySearch()` — привязывает entry к строке буфера через поиск

**Нужно связать всё вместе:**

## Текущий флоу (проблемы)

## Предлагаемый новый флоу

### Фаза 1: Инициация (Cmd+T)
- Создаём проект в БД (как сейчас, с `__unset__` path)
- `openProject()` открывает его с **`currentView: 'home'`** (не terminal)
- Терминал НЕ создаётся на этом этапе — экономим ресурсы, нет пустого таба
- Проект в **"draft" режиме** (отслеживаем через `lastCreatedProjectId`, уже есть)

### Фаза 2: Home View в Draft Mode
На Home view рендерится специальный "creation layout":

```
┌──────────────────────────────────────────┐
│                                          │
│  [  Новый проект___________  ]  (400px)  │  ← инпут, auto-focus, текст выделен
│                                          │
│  📁 No directory selected  [📝]          │  ← кнопка подсвечена accent-цветом
│                                          │
│  (пусто — нет табов, нет истории)        │
│                                          │
└──────────────────────────────────────────┘
```

- Инпут ~400px, auto-focused, текст "Новый проект" pre-selected
- Кнопка "выбрать директорию" (карандаш) подсвечена accent-цветом — визуально приглашает нажать
- Секции "Active Tabs", "Favorites", "History" скрыты (или показывают пустое состояние)

### Фаза 3: Действия пользователя

**Сценарий A — Ввёл имя, нажал Enter:**
- Имя сохраняется, проект выходит из draft mode
- `lastCreatedProjectId = null`
- Создаётся Terminal 1, переключается на `currentView: 'terminal'`
- Директория остаётся `__unset__` (можно сменить позже)

**Сценарий B — Кликнул на кнопку директории:**
- Тут ловушка: `onBlur` инпута стрельнёт ДО `onClick` кнопки
- Решение: `onMouseDown` на кнопке ставит флаг `isPickingDirectory = true`
- `onBlur` инпута проверяет флаг → если `true`, НЕ отменяет создание
- Открывается системный диалог выбора папки
- Если пользователь выбрал папку:
  - `path` ставится на выбранную директорию
  - `name` автоматически = последний сегмент пути (e.g. `/Users/foo/my-app` → "my-app")
  - Инпут обновляется с новым именем
  - Проект **подтверждается** (выход из draft mode)
  - Создаётся Terminal 1, переключается на terminal view
- Если пользователь отменил диалог:
  - Возвращаемся на Home, всё ещё в draft mode
  - Фокус обратно на инпут имени

**Сценарий C — Escape:**

- Удаляем проект из БД

**Сценарий D — Blur (кликнул мимо):**

- Удаляем проект, возвращаемся назад

## Текущая архитектура (уже в коде)

```

  │

  │     │                      
  │     └─► updateViewport()   ──► viewportStates.set()
  │                                    │
  │                                    └─► viewportCallback() ──► debounce 100ms
  │                                                                    │
  │                                                              checkVisibility()
  │                                                                    │
  │                            getVisibleText(tabId) ◄─────────────────┤
  │                              │                                     │
  │                              ├─ buf.viewportY                      │
  │                              ├─ buf.viewportY + rows               │
  │                              └─ getLine(i).translateToString() ──► visibleText
  │                                                                    │

  │                                                                entry.content

  │                                                                    │

  │                                                                    │
  │                                                              dot: 5px, 0.75 opacity, glow
```

## Почему именно так, а не через markers

Были два варианта:

**Вариант A (markers)** — `bindEntryBySearch()` для каждого entry при загрузке, потом проверка `marker.line ∈ [viewportTop, viewportBottom]`:
- Проблема: `bindEntryBySearch` вызывает `findNext()` → **уничтожает user selection** в терминале
- Проблема: нужно бидить ВСЕ entries при каждом `loadTimeline` (каждые 2 сек) — дорого
- Проблема: markers становятся `disposed` когда scrollback trimming удаляет старые строки

**Вариант B (прямое чтение буфера)** — то что сделано:
- `getVisibleText()` читает `~50 строк × ~200 chars = 10KB` напрямую из `buffer.getLine()`
- Для `~30 entries` проверяет `includes()` 40-символьного snippet
- **Не трогает selection**, не дёргает `findNext`, не создаёт markers
- Дебаунс 100ms → при бешеном скролле не тормозит

## Ограничения текущего подхода

1. **False positives** — если пользователь два раза написал одинаковые первые 40 символов, оба кружочка подсветятся. Но это же ограничение есть и у `searchAndScroll` при клике.

2. **Не ловит "почти видно"** — entry текст может быть на 1 строку выше viewport, и подсветки не будет. Можно расширить окно на ±5 строк, но это усложнение без явной пользы.

3. **Частота обновления** — viewport callback приходит на каждый `onScroll`, `onWriteParsed` (десятки раз в секунду при активном output). Дебаунс 100ms ограничивает до ~10 проверок/сек максимум.
