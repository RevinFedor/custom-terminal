⏺ Теперь у меня полная картина. Вот анализ принципиальных различий:

  ---
  Сравнение систем сохранения сессий: Claude vs Gemini

  Gemini CLI — "Trojan Horse" (Прямая инъекция)

  Как хранит сессии CLI:
  - Checkpoints сохраняются в ~/.gemini/tmp/<SHA256_HASH>/checkpoint-*.json
  - SHA256 вычисляется от пути к проекту
  - CLI не имеет внутреннего реестра — просто сканирует папку

  Механизм восстановления:
  1. Вычислить SHA256 директории проекта
  2. Напрямую записать JSON-файл в нужную папку Gemini
  3. Пользователь делает /chat resume <name>

  Плюс: Полностью невидимая инъекция — никакой "грязи" в терминале.

  Детекция готовности CLI (важно!):
  - Primary: Детекция скрытия курсора (\x1b[?25l) — Gemini/Ink прячет курсор
   когда готов к вводу
  - Fallback: Тишина в PTY-потоке 1.5-2 сек

  ---
  Claude Code — "Sniper Watcher" (Наблюдение за ФС)

  Как хранит сессии CLI:
  - Сессии в ~/.claude/projects/<project_path>/<UUID>.jsonl
  - UUID генерируется Claude при старте
  - Проблема: Claude валидирует JSON при --resume — пустой/битый файл =
  отказ

  Неудачные попытки инъекции (документировано):
  1. ❌ Создать пустой .jsonl с нашим UUID → Claude считает файл
  поврежденным
  2. ❌ Использовать agent-* префиксы → Триггерит TUI-picker меню
  3. ❌ Автоматический Enter через 1 сек → Слишком хрупко

  Итоговое решение — Sniper Watcher:
  1. При вводе claude → активируется fs.watch на 5 сек
  2. Ловится событие создания файла *.jsonl с форматом UUID
  3. UUID сохраняется в store и БД
  4. Теперь можно делать claude --resume <uuid> и форки

  ---
  Краткий вывод:

  Claude — UUID-based система. Зная UUID, можно продолжить сессию откуда
  угодно. Сложность только в захвате этого UUID при старте (решено Sniper
  Watcher).

  Gemini — Path-based система. Checkpoint привязан к SHA256 директории.
  Перенос между проектами требует пересчёта хеша и физического копирования
  файла в другую папку. Это значительно сложнее автоматизировать для
  cross-project форков.

  Поэтому да — для Claude перенос между вкладками реально делается "по
  щелчку" (просто --resume UUID), а для Gemini нужно патчить пути в JSON и
  перекладывать файлы между папками с разными хешами.

  ---
  Как это работает:

  1. Ввод "gemini" → перехват → Sniper Watcher активирован
  2. Gemini создаёт файл session-*.json в ~/.gemini/tmp/<hash>/chats/
  3. Watcher ловит файл → парсит JSON → получает sessionId
  4. sessionId сохраняется в tab → персистится в БД
  5. gemini-c / gemini-f используют сохранённый ID

  Теперь сессии Gemini сохраняются при закрытии терминала и
  восстанавливаются при следующем открытии — аналогично Claude!
