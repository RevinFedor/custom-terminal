# Отчет по разработке механизма интеграции с Rewind (Claude Code TUI)

## 1. Изначальная цель
Реализовать программное управление встроенным меню `Rewind` в терминале Claude Code. Задача: автоматизировать открытие меню истории, парсинг списка доступных точек восстановления (fork markers), навигацию к нужной точке с помощью стрелок и последующий вызов команды восстановления.

## 2. Структура файлов
*   **`src/main/main.js`**: Логика IPC-хендлера `claude:open-history-menu`. Здесь сосредоточен весь низкоуровневый ввод-вывод PTY.
    *   *Почему так:* Взаимодействие с PTY требует доступа к объекту терминала, который находится в основном процессе. Здесь же реализован стейт-машина для отслеживания ответов терминала.
*   **`src/renderer/components/Workspace/Timeline.tsx`**: Интерфейсная часть. Вычисляет `targetIndex` на основе выбранного пользователем события в UI-таймлайне.
    *   *Почему так:* Индекс сообщения должен соответствовать только пользовательским записям (`type === 'user'`), фильтруя системные уведомления, чтобы индекс в UI совпал с индексом в TUI терминала.

## 3. Краткий отчет: что работает
1.  **Мгновенный захват меню**: Открытие через `Ctrl+C` -> `Esc` -> `Esc` с детекцией готовности через маркеры синхронизированного вывода.
2.  **Парсинг TUI**: Корректное извлечение текста сообщений с восстановлением пробелов.
3.  **Точная навигация**: Перемещение курсора `❯` на заданное количество позиций вверх с верификацией каждого шага.
4.  **Синхронизация**: Полный отказ от фиксированных таймеров в пользу событийного ожидания ответа терминала.

## 4. Детальный разбор проблем и неудачных решений

Ниже перечислены критические моменты, где нейросеть ошибалась в логике или не обладала специфическими знаниями о работе TUI Claude Code. **Важно внести эти правки в основной файл архитектуры взаимодействия с терминалом.**

### Итерация 1: Ошибка стратегии ожидания (Таймеры vs Сигналы)
*   **Что не сработало:** Использование `setTimeout(3000)` для ожидания отрисовки меню.
*   **Причина ошибки:** TUI терминала отрисовывается неравномерно. Таймер либо слишком длинный (задержка), либо слишком короткий (неполный буфер).
*   **Решение (Fix):** Использование маркера синхронизированного вывода `\x1b[?2026l`.
*   **Заметка для доки:** Claude Code (и современные терминалы) используют этот маркер для обозначения конца кадра отрисовки. Нейронка должна всегда проверять наличие `\x1b[?2026l` в буфере перед началом парсинга.

### Итерация 2: Проблема исчезновения пробелов (VT-последовательности)
*   **Что не сработало:** Использование стандартного `stripVTControlCharacters` для очистки буфера.
*   **Причина ошибки:** Claude Code TUI не использует символ пробела (`0x20`) для разделения слов. Вместо этого он использует последовательность `\x1b[NC` (cursor forward). Стандартный стриппер удаляет эти команды, склеивая слова в "каксейчасработает".
*   **Решение (Fix):** Добавлен пре-процессинг буфера: замена `\x1b[(\d*)C` на соответствующее количество пробелов перед удалением остальных VT-символов.
*   **Заметка для доки:** При парсинге TUI-интерфейсов нельзя полагаться на стандартные методы очистки текста, если важна структура строк.

### Итерация 3: Ошибка парсинга Diff-рендеринга (Частичное обновление)
*   **Что не сработало:** Повторный запуск полного парсера меню после нажатия стрелки "Вверх".
*   **Причина ошибки:** При нажатии стрелки терминал не перерисовывает все меню целиком. Он присылает только "diff" — последовательности перемещения курсора и обновления только тех строк, которые изменились. Полный парсер не находил в этом куске текста заголовок или список всех записей.
*   **Решение (Fix):** Функция `findCursorInDiff`. Мы сохраняем список записей (`knownEntries`) при первом открытии, а после стрелок ищем маркер `❯` в пришедшем диффе и сопоставляем текст рядом с ним с нашим списком по префиксу.
*   **Заметка для доки:** Навигация в TUI должна быть разделена на фазу "Инициализация" (полный парсинг) и фазу "Слежение" (сопоставление текущей строки с ранее известными).

### Итерация 4: Ошибка регулярного выражения разделителя
*   **Что не сработало:** Регулярка `/^[─]+\$/` для фильтрации декоративных линий.
*   **Причина ошибки:** Опечатка нейронки — экранирование знака доллара `\так же ниже скину одработки -  заставляло искать литерал доллара в конце линии, вместо конца строки `так же ниже скину одработки - . Разделители попадали в список сообщений как пустые записи.
*   **Решение:** Исправлено на `/^[─]+$/`.

вот отчет ресеча # Why bracketed paste + Enter fails in Ink TUIs via node-pty

**The root cause is a convergence of three independent bugs, not one.** Your `\r` is not being "swallowed" by a single mechanism — rather, a PTY buffer split on macOS breaks the bracketed paste escape sequence, Ink lacks a paste-aware state machine so it can't reassemble it, and React's asynchronous state batching means Enter sees stale (empty) input state even when paste does arrive intact. Additionally, waiting for `\x1b[?2026l` as a readiness signal is fundamentally misguided — it is an output rendering protocol that has nothing to do with input processing. The most reliable fix is to bypass the Ink TUI entirely using `claude -p` (headless mode) or the Claude Agent SDK.

## macOS will split your 2023-byte write at the kernel level

**PIPE_BUF atomicity guarantees do not apply to PTY devices.** PIPE_BUF (4096 bytes on Linux, 512 bytes on macOS) is a POSIX guarantee for pipes and FIFOs only. PTYs are character devices governed by entirely different buffering in the TTY line discipline layer.

On macOS, the critical limit is **TTYHOG = 1024 bytes** — the raw input queue size defined in XNU's `bsd/sys/tty.h`. Your 2023-byte payload is roughly **2× this limit**. The kernel physically cannot hold the entire write in the PTY input buffer at once. The child process will receive the data across multiple `read()` calls, and the split point is unpredictable. This is not theoretical: node-pty PR #831 (merged December 2025) explicitly confirms *"pasting large amounts of data (1-4KB, but macOS seems to be 1KB)"* causes interleaving and data loss. The Kitty terminal's issue #5869 and VS Code's issue #38137 independently confirm data corruption at the 1024-byte boundary on macOS.

On Linux, **N_TTY_BUF_SIZE = 4096 bytes**, so your 2023-byte payload fits within the buffer. However, there is still no guarantee of atomic delivery — `read()` may return fewer bytes than available (a "short read"). The probability of a split is moderate on Linux but **near-certain on macOS**.

The practical consequence: your bracketed paste sequence `\x1b` + 2011 chars + `\x1b[201~` arrives to the child process as fragments. The `\x1b[201~` paste-end marker may land in a different `read()` chunk than the paste content, or worse, the 6-byte sequence `\x1b[201~` itself may be **split across reads** (e.g., `\x1b[20` in one read, `1~` in the next). The micro text editor's issue #3668 documents exactly this: `\x1bissues\x1b[20` in one read followed by `1~` in the next — direct evidence of escape sequence splitting on PTY devices.

## Ink has no bracketed paste state machine

Ink's input architecture processes raw stdin bytes through a pipeline: `stdin data event → parseKeypress() → useInput callback`. The critical finding is that **Ink has no awareness of bracketed paste mode at all**. It does not track a "currently inside paste" state, does not recognize `\x1b` as a paste-start marker, and does not buffer input until `\x1b[201~` arrives.

When `parseKeypress()` encounters `\x1b`, it parses it as a CSI (Control Sequence Introducer) escape sequence. But the behavior depends entirely on whether the sequence arrives in one piece or fragmented. If the 6-byte `\x1b` sequence is split across `data` events (possible with the macOS buffer split described above), the parser sees an incomplete escape sequence fragment, which may be:

- Dropped silently as an unrecognized escape
- Interpreted as separate characters (literal `~`, digits, etc.)
- Merged incorrectly with subsequent data into a malformed sequence

The `useInput` hook contains a revealing line: `if (input.startsWith('\u001B')) { input = input.slice(1); }` — this strips a leading ESC byte from input, confirming that escape sequence handling is rudimentary. There is no buffering between `data` events for incomplete escape sequence reassembly.

Each parsed keypress is dispatched individually through `reconciler.batchedUpdates(() => { inputHandler(input, key); })`. This means that even when paste text and `\r` arrive in the same `data` chunk, they are processed as separate keypress events within React's batching context — which leads directly to the state race condition.

## React's batched state updates create a paste-then-enter race

This is the most insidious of the three bugs. When paste text and `\r` arrive in rapid succession (or in the same `data` event), the processing sequence is:

1. `parseKeypress(buffer)` returns `[pasteKeypress, enterKeypress]`
2. Paste keypress dispatched → `onChange(pastedText)` calls `setState(pastedText)` — **queued, not committed**
3. Enter keypress dispatched → `onSubmit(value)` reads `value` from **current render's props** — this is still the **pre-paste value** (empty string)
4. React eventually processes all batched updates and re-renders — but `onSubmit` already fired with stale state

React 18's automatic batching makes this worse, not better. State updates from multiple `batchedUpdates()` calls within the same microtask are coalesced into a single render. The re-render that would make the pasted text available as `value` happens **after** all input events in the current tick have been processed. For programmatic input where paste and Enter arrive with sub-millisecond timing, this race is virtually guaranteed.

**This is confirmed by Claude Code issue #2552**: *"When I paste long text and press Enter too quickly after pasting, it will freeze Claude Code."* Issue #15553 goes further, documenting that programmatic `\r`/`\n` sent via `terminal.sendText()` **never triggers onSubmit** — it is treated as a literal newline character rather than a submit action. Multiple approaches were tried (sendText, sendSequence with `\r`, `\n`, `\x0d`, `\x0a`, AppleScript keystrokes, macOS key codes) and all failed.

## The synchronized output marker is the wrong signal entirely

**Synchronized Output (mode 2026) is an OUTPUT rendering protocol. It has zero connection to input processing.** The sequences `\x1b[?2026h]` (begin synchronized update) and `\x1b[?2026l]` (end synchronized update) flow from the TUI application to the terminal emulator to prevent screen tearing during frame updates. They wrap visual output (cursor moves, text drawing, color changes), not input events.

Ink emits `\x1b[?2026h` before writing a frame to stdout and `\x1b[?2026l]` after completing the frame write. Analysis of Claude Code's terminal output confirms **100% of visible output is wrapped in sync blocks**, alternating between incremental updates (~2.7 KB average) and full redraws (~94.5 KB average).

The reason `\x1b[?2026l` is not received after paste is simple: **a paste event does not necessarily trigger a render**. If the pasted text causes a React state update but Ink's 32ms render throttle has not yet fired, or if the state update hasn't caused a visible change, no sync block will be emitted. Waiting for `\x1b[?2026l` as a "paste processing complete" signal will result in waiting indefinitely — exactly the 5-second timeout you observe.

In your Electron + node-pty architecture, the Electron app is the terminal emulator. Mode 2026 sequences arrive as raw bytes via `onData`. They are purely rendering hints for your terminal emulation layer and carry no information about the application's input readiness state.

## Claude Code has a documented history of paste bugs

The Claude Code repository contains at least six paste-related issues, establishing a pattern of fundamental fragility in the Ink-based input pipeline:

- **#2552** — Paste + rapid Enter causes application freeze (closed as duplicate)
- **#13183** — CLI hangs on paste from accessibility/voice-to-text tools; `\x1b[201~` appears as raw text in prompt, proving the paste parser is failing

- **#4772** — Pasting content causes permanent CLI freeze (v1.0.63)

Issue #15553 is the definitive statement on programmatic input: it is a **feature request** for programmatic input submission support, explicitly documenting that this capability does not exist in the current architecture.

## Recommended solutions in priority order

**Use `claude -p` (headless/print mode).** This completely bypasses the Ink TUI and its entire class of input race conditions. It reads prompts from arguments or stdin, executes, and prints results:

```javascript
// Structured JSON output, no PTY needed
const result = spawn('claude', ['-p', prompt, '--output-format', 'stream-json']);

// Multi-turn via session resume  
spawn('claude', ['-p', '--resume', sessionId, '--output-format', 'stream-json']);

// Streaming JSON input for multi-turn without relaunching
spawn('claude', ['-p', '--input-format', 'stream-json', '--output-format', 'stream-json']);
```

**Use the Claude Agent SDK** for TypeScript or Python. It spawns Claude CLI as a subprocess with structured JSON I/O over stdin/stdout pipes — no PTY involved.

**If PTY interaction is truly required**, use output-based synchronization (the "expect" pattern) rather than timing-based approaches. Send the bracketed paste, wait for the application to echo or render a distinctive substring of the pasted text in its output, add a **200ms+ delay** for React to commit state, then send `\r`. Scale the delay with payload size: `Math.max(200, payload.length * 0.15)` ms as a baseline heuristic. Never include `\r` inside bracketed paste delimiters — it will be treated as literal text, not as a submit action.

## Conclusion

The bug is not a single race condition but a **three-layer failure**: macOS kernel PTY buffer limits split the bracketed paste escape sequence at ~1024 bytes, Ink's parser lacks state to reassemble fragmented paste sequences, and React's batched state updates ensure that even when paste arrives intact, Enter fires before the pasted text is committed to state. The synchronized output protocol is a red herring — it governs terminal rendering, not input readiness. The Ink TUI was never designed for programmatic input at this scale, as confirmed by Claude Code's own issue tracker. The `claude -p` headless mode exists precisely for this use case and eliminates all three failure modes at once.

## Архитектура без таймаутов: Chunked Paste + Echo Verification

### Почему таймауты — плохо

Таймаут — это гадание. 200ms может быть мало при нагрузке, и слишком много в нормальном режиме. Нам нужна **event-driven синхронизация**.

### Ключевой инсайт

Когда Ink TUI получает текст и рендерит его в input box, этот рендер летит в stdout → PTY → наш `onData`. **Если мы видим echo — значит React уже отрендерил, значит state уже committed.** Это и есть наш сигнал — не таймер, а факт.

### Алгоритм

```
Payload: 5000 байт текста

1. Разбить на чанки по ~900 байт (< TTYHOG 1024, минус 12 байт на paste brackets)

2. Для каждого чанка:
   ├─ term.write(\x1b[200~ + chunk + \x1b[201~)   // atomic write < 1024
   ├─ marker = последние 20 символов чанка
   └─ ЖДАТЬ: слушать onData → stripVT → includes(marker)?
      ├─ ДА → чанк принят, Ink отрендерил, state committed → следующий чанк
      └─ 5s safety timeout → abort (что-то сломалось)

3. Все чанки подтверждены → term.write('\r')
```

### Почему это работает

| Проблема из ресёрча | Как решается |
|---|---|
| **TTYHOG 1024 bytes** | Каждый write < 1024 — ядро не режет |
| **Ink не клеит фрагменты** | Каждый чанк — полный `\x1b[200~...~\x1b[201~` |
| **React batching race** | `\r` шлём ПОСЛЕ echo = ПОСЛЕ render = state committed |
| **Sync markers бесполезны** | Не используем. Echo текста = наш сигнал |
| **Любая длина** | Чанков может быть сколько угодно |

### Ink и множественные paste

Ink обрабатывает каждый bracketed paste как "вставить текст в cursor position". Несколько paste подряд → текст конкатенируется в input. Это штатное поведение — так работает любой paste в любом редакторе.

### Почему delay после echo НЕ нужен

Echo в output означает полную цепочку:
1. Ink получил paste data
2. React `setState(pastedText)` 
3. React отрендерил (иначе откуда echo?)
4. Ink записал рендер в stdout
5. Мы получили через onData

К моменту echo state **гарантированно committed**. `onSubmit(value)` прочитает правильное значение.

### Safety timeout — не механизм, а fallback

5 секунд — это не "подождать на всякий случай", а "если за 5 секунд echo не пришел — что-то фатально сломалось, abort и логируем ошибку". В нормальном режиме echo приходит за 30-100ms.

## Фикс: sync marker вместо text echo

**Проблема:** Ink TUI коллапсирует вставленный текст в `[Pasted text #1 +11 lines]` — пастнутый текст НИКОГДА не появляется в PTY output. `waitForEcho` с текстовым маркером ждал 8 секунд, не находил, fallback через timeout → 8s + 8s + 0.1s = ~16 секунд.

**Решение:** Заменил `waitForEcho` (ищет текст) на `waitForRender` (ищет `\x1b[?2026l` sync marker). Sync marker работает потому что:
1. Чанк < 1024 bytes → ядро НЕ режет → Ink получает целиком
2. Ink обрабатывает paste → React state update → re-render → sync marker
3. Sync marker = render завершён = state committed

**Ожидаемый результат:** вместо `[8007, 8001, 113]ms` должно быть `[~50, ~50, ~50]ms` — sync marker приходит за 30-100ms (Ink render throttle = 32ms).
