## 2. Тройной Обязательный Фильтр (Entry Point)

AI **ОБЯЗАН** прочитать три базовых файла в начале сессии:

| Файл | Роль | Отвечает на вопрос |
|------|------|-------------------|
| `context.md` | Точка входа | "Что это за проект?" |
| `architecture.md` | Техническое устройство | "Как это работает внутри?" |
| `main-feature.md` | Пользовательские сценарии | "Что видит и делает юзер?" |

**Шаблон context.md:**

```markdown
# Context: [Project Name]

> **ИНСТРУКЦИЯ (System Prompt):**
> 1. В начале сессии ОБЯЗАТЕЛЬНО прочитай три базовых файла:
>    - `context.md` — точка входа (этот файл)
>    - `architecture.md` — технический фундамент
>    - `main-feature.md` — пользовательские сценарии
> 2. **ПЕРЕД ИЗМЕНЕНИЕМ КОДА** — найди нужную фичу в `main-feature.md`, перейди в `features/` и прочитай связанные `knowledge/`.
> 3. Если есть ссылка на `knowledge/` — ЧИТАЙ ОБЯЗАТЕЛЬНО (там причины критических решений).
> 4. Игнорируй папки с пометкой ⚪ (tmp, journal) без команды.

## 1. Обзор Проекта
(Краткая суть продукта)

## 2. Технический Стек
(Основные технологии со ссылками на knowledge/ где нужно)

## 3. Карта Документации
- **Главная (Flows):** `main-feature.md`
- **Код (Архитектура):** `architecture.md`
- **Инфраструктура (Ops):** `infrastructure/`
- **База Знаний:** `knowledge/`
```

## 3. Главная карта — main-feature.md

Файл `main-feature.md` живёт в **корне docs/** (не в features/).

### Правила:
1. **Все фичи упомянуты** — каждый файл из `features/` должен иметь ссылку здесь.
2. **User Flow стиль** — описание через сценарий пользователя, не сухие факты.
3. **Без кода** — никаких компонентов и функций. Только ссылки на детальные файлы.

### Пример:

```markdown
# Main Features: User Flows

## 1. Управление табами
**Flow:** Пользователь нажимает `Cmd+T` → создаётся таб с именем `tab-1` →
вводит `npm run dev` → таб переименовывается в `run-dev` и становится зелёным.

→ Детали: `features/tabs.md`
→ Почему так: `knowledge/fix-ui-stability.md`

## 2. AI Research
**Flow:** Пользователь выделяет текст → нажимает "Research" →
открывается панель с Gemini → получает анализ.

→ Детали: `features/research.md`
```

## 4. Правило "Ни одного брошенного файла" (Full Coverage)

### Knowledge Audit
Каждый файл в `knowledge/` **ДОЛЖЕН** быть упомянут (иметь ссылку) в:
- `architecture.md` — если это про технику/инфраструктуру
- Конкретной фиче в `features/` — если это про логику фичи

### Feature Audit
Каждый файл в `features/` **ДОЛЖЕН** быть упомянут в `main-feature.md`.

### Зачем
Если файл не упомянут в основных путеводителях, AI может его пропустить → нарушение логики или повторное изобретение велосипеда.

---

## 5. Принцип "Что/Как" vs "Почему"

| Уровень | Отвечает на | Где живёт |
|---------|-------------|-----------|
| **Что это?** | Описание функционала | `main-feature.md`, `features/` |
| **Как работает?** | Техническая реализация | `architecture.md`, `features/code-map` |
| **Почему так?** | Причины решений, баги, ограничения | `knowledge/` |

### Связь уровней
В описании Flow или архитектуры **обязательно** ссылка на `knowledge/`, если там описан:
- Костыль или workaround
- Причина выбора технологии (Canvas вместо WebGL)
- Внешнее ограничение API

---

## 6. Динамическое обновление (Knowledge Traceability)

Любое новое исправление или технический факт, найденный в ходе сессии:

1. **Оформляется** как новый файл в `knowledge/` (fact- или fix-)
2. **Сразу прописывается** ссылкой в соответствующем разделе:
   - `architecture.md` — если про технику
   - `features/*.md` — если про конкретную фичу
3. **Записываются неудачные попытки** — что пробовали и почему не сработало

## 7. Философия разделов

### 🟢 Features (это ТОЛЬКО то, что мы создали САМИ)
Источник истины — **ВНУТРИ** проекта. Наши решения, правила, логика.
*Пример:* "Smart Restart кнопка в табах" → `features/tabs.md`

### 🟢 Knowledge (это ТОЛЬКО то, что пришло ИЗВНЕ)
Источник истины — **ВОВНЕ**. Внешние ограничения, факты, проблемы.
*   **ФАКТ (`fact-`):** "Мир устроен так". Внешние данные (API, протоколы).
*   **ОПЫТ (`fix-`):** "Мы пробовали — сломалось". Решения проблем.

### 🟢 Infrastructure (это КАК запустить проект)
Источник истины — **ОПЕРАЦИИ**. Команды терминала, пути, адреса.

**Критерий:** Если можно выполнить в терминале или это путь/адрес → infrastructure.

**НЕ infrastructure:**
*   Хоткеи (это UI фича → `features/`)
*   Цвета/стили UI (это UI фича → `features/`)
*   Глобальное состояние (это архитектура → `architecture.md`)

---

## 8. Структура Фичи (Принцип Матрёшки)

Любая фича подразделяется на **максимум 3 блока**:

1. **Intro + User Flow:** Что это? Зачем? Как пользователь взаимодействует?
2. **Behavior Specs:** Функциональные особенности. Чем отличается от стандарта?
3. **Code Map:** Техническая навигация. Файлы, функции, переменные.

---

## Итог для AI

Когда агент заходит в проект:
1. Читает **Тройку**: context → architecture → main-feature
2. Из `main-feature.md` понимает какой **Flow** будет менять
3. Из `architecture.md` понимает какие **IPC/БД** затронет
4. По ссылкам идёт в `knowledge/` чтобы **не наступить на грабли**

## 5. Правила Редактирования Документации

  ### ⚠️ КРИТИЧНО: Дополняй, не перезаписывай

  При обновлении существующих файлов:

  1. **Сначала ПРОЧИТАЙ весь файл** — не редактируй по памяти.
  2. **ДОБАВЛЯЙ новую информацию** к существующей, не заменяй блоки целиком.
  3. **Если удаляешь что-то** — явно обоснуй почему (устарело, неверно, дублируется).

  **Плохо:**
  - - **Main Process:** Управляет `node-pty`, SQLite и файловой системой.
  + - **Main Process:** Управляет `node-pty` и JSON-сторами.
  (Потеряли SQLite и файловую систему)

  Хорошо:
  - - **Main Process:** Управляет `node-pty`, SQLite и файловой системой.
  + - **Main Process:** Управляет `node-pty`, SQLite, JSON-сторами и файловой системой.
  (Добавили JSON-сторы, сохранили остальное)

## 6. Формат Плана Перед Редактированием

  При описании плана изменений указывай ТРИ категории:

  ### ➕ ДОБАВЛЯЕТСЯ (новое)
  - `tabs.md` → Behavior: добавить "Цветовая маркировка"
  - `knowledge/` → новый файл `fix-tab-persistence.md`

  ### ✏️ ИЗМЕНЯЕТСЯ (было → стало)
  - `architecture.md` → секция "Data Layer": добавить поле `color` в описание JSON

  ### 🗑️ УДАЛЯЕТСЯ (с обоснованием)
  - `tabs.md` → убрать устаревшую ссылку на `old-dnd.md` (файл удален)

  **Уровень детализации:**
  - Указывай СЕКЦИИ файла (Intro, Behavior, Code Map), не весь файл
  - Указывай НАЗВАНИЯ функций/методов только для Code Map
  - НЕ пиши код — только имена и краткое описание

  **Пример хорошего плана:**
  tabs.md:
    ➕ Behavior: "System Tools (зеленый цвет)"
    ➕ Code Map: ссылка на fix-tailwind-dynamic-runtime.md
    ✏️ Intro: расширить описание (добавить про цвета, сохранить про зоны)

## 7. Порядок Работы (ОБЯЗАТЕЛЬНО)

### ШАГ 1: Сначала ПРОЧИТАЙ ВСЮ существующую документацию
Перед анализом сессии и составлением плана — **прочитай ПОЛНОСТЬЮ текущее состояние docs/**:
1. `docs/context.md` — точка входа
2. `docs/features/main.md` — карта модулей
3. `docs/architecture.md` — текущая архитектура
4. **ВСЕ файлы `docs/features/*.md`** — прочитай содержимое каждого
5. **ВСЕ файлы `docs/knowledge/*.md`** — прочитай содержимое каждого

**⚠️ КРИТИЧНО:** Не просто список файлов, а именно СОДЕРЖИМОЕ. Ты должен знать что написано внутри, прежде чем планировать изменения.

**Зачем:** Чтобы понимать что УЖЕ есть, и не потерять/не продублировать информацию.

### ШАГ 2: Анализ сессии → План
Только после прочтения docs/ — анализируй файл сессии и составляй план (➕/✏️/🗑️).

### ШАГ 3: Одобрение → Редактирование
После одобрения плана — редактируй файлы, **сверяясь с тем что прочитал в ШАГ 1**.
