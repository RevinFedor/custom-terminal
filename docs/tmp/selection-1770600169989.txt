# Как на самом деле работает `/compact` в Claude Code

## Короткий ответ

Нейронка в приложенном файле **ошибалась**, когда говорила что compact "полностью стирает сессию". `/compact` **не удаляет** историю — он **суммаризирует** её и создаёт новый контекст на основе саммари. А то, что ты видишь историю до compact при `claude -c`, — это **ожидаемое поведение** системы экспорта и отображения session tree.

---

## Что делает `/compact` на уровне JSONL

Из анализа логов в твоём файле видно, что при compact Claude Code делает следующее внутри JSONL-файла сессии:

1. **Создаёт запись `compactboundary`** — системная запись типа `{type: "system", subtype: "compactboundary"}` с полем `logicalParentUuid`, указывающим на UUID последней записи до compact.
2. **Создаёт user-сообщение с саммари** — запись типа `user` с текстом `"This session is being continued from a previous conversation that ran out of context. The summary below covers the earlier portion..."` и дальше идёт само саммари длиной ~6000-8000 символов.
3. **Старые записи остаются в JSONL** — файл append-only, ничего не удаляется. Все предыдущие записи (user, assistant, tool results) физически остаются на диске.

Из лога (цитата из твоего файла):

```
Entry types: user 20, assistant 14, progress 6, system/compactboundary 2
Total: 42 entries
Summary entries: 0
Compact boundary logicalParentUuid: a9874018
```

То есть в файле `2a05b7b5.jsonl` живут **42 записи**, из которых 2 — это compact boundaries. Записи до compact **не удалены**.

---

## Почему `claude -c` показывает историю ДО compact

Когда ты запускаешь `claude -c` (или `--continue`), Claude Code:

1. **Строит session chain** — проходит по цепочке JSONL-файлов через bridge entries (`resolveSessionChain`). Например, из лога: `Chain depth 2`, `Chain depth 3` — это значит, что он объединяет 2-3 файла сессий.

2. **Делает merge records** — сливает все записи из цепочки в единый список. Например: `Merged records 107`, `Chain depth 3`, `SessionId distribution: 2a05b7b5(14), 4b13df35(22), 05922cf0(5)` — это 3 сессии в одной цепочке.

3. **Делает backtrace** — идёт по `parentUuid` от последней записи назад к началу. И вот тут ключевой момент: **backtrace обрывается на compact boundary**, потому что `logicalParentUuid` ссылается на UUID, который не существует в текущей ветке (это UUID из записей ДО compact, которые были в предыдущей сессии).

Из лога:
```
Backtrace ended - UUID not found: a9874018-5ab8-4172-b6cc-e688fab794fe
```

4. **Строит Session Tree для отображения** — и показывает дерево сегментов:
```
Session Tree:
  2a05b7b5 (root) — 7 prompts, 2 tools
  4b13df35 (fork) — 2 prompts, 5 tools
  05922cf0 (plan mode) — 2 prompts, 1 tool
```

Записи ДО compact отображаются в дереве как **root-сегмент**, потому что они физически присутствуют в JSONL-файлах. Но **в контекст модели** они не попадают — модель получает только саммари.

---

## Различия: `/compact` vs `/clear`

| Аспект | `/compact` | `/clear` |
|--------|-----------|----------|
| Контекст модели | Очищается, заменяется саммари | Полностью очищается, ничего нет |
| JSONL-файл | Записи остаются, добавляется `compactboundary` + саммари | Контекст сбрасывается |
| Session chain | Продолжает ту же цепочку сессий | Та же сессия, но пустой контекст |
| При `--continue` | Видна вся история в Session Tree (root → fork → plan mode...) | — |
| Знание о прошлом | Есть через саммари | Нет |

Это ключевое различие, которое нейронка в файле не учла:

- **Ты при `claude -c`** видишь **Session Tree** — это инструмент экспорта/визуализации, который показывает полное дерево сессий из JSONL-файлов. Он специально показывает root-сегмент со старыми промптами, чтобы ты видел историю.

# Как на самом деле работает `/compact` в Claude Code

## Короткий ответ

Нейронка в приложенном файле **ошибалась**, когда говорила что compact "полностью стирает сессию". `/compact` **не удаляет** историю — он **суммаризирует** её и создаёт новый контекст на основе саммари. А то, что ты видишь историю до compact при `claude -c`, — это **ожидаемое поведение** системы экспорта и отображения session tree.

---

## Что делает `/compact` на уровне JSONL

Из анализа логов в твоём файле видно, что при compact Claude Code делает следующее внутри JSONL-файла сессии:

1. **Создаёт запись `compactboundary`** — системная запись типа `{type: "system", subtype: "compactboundary"}` с полем `logicalParentUuid`, указывающим на UUID последней записи до compact.
2. **Создаёт user-сообщение с саммари** — запись типа `user` с текстом `"This session is being continued from a previous conversation that ran out of context. The summary below covers the earlier portion..."` и дальше идёт само саммари длиной ~6000-8000 символов.
3. **Старые записи остаются в JSONL** — файл append-only, ничего не удаляется. Все предыдущие записи (user, assistant, tool results) физически остаются на диске.

Из лога (цитата из твоего файла):

```
Entry types: user 20, assistant 14, progress 6, system/compactboundary 2
Total: 42 entries
Summary entries: 0
Compact boundary logicalParentUuid: a9874018
```

То есть в файле `2a05b7b5.jsonl` живут **42 записи**, из которых 2 — это compact boundaries. Записи до compact **не удалены**.

---

## Почему `claude -c` показывает историю ДО compact

Когда ты запускаешь `claude -c` (или `--continue`), Claude Code:

1. **Строит session chain** — проходит по цепочке JSONL-файлов через bridge entries (`resolveSessionChain`). Например, из лога: `Chain depth 2`, `Chain depth 3` — это значит, что он объединяет 2-3 файла сессий.

2. **Делает merge records** — сливает все записи из цепочки в единый список. Например: `Merged records 107`, `Chain depth 3`, `SessionId distribution: 2a05b7b5(14), 4b13df35(22), 05922cf0(5)` — это 3 сессии в одной цепочке.

3. **Делает backtrace** — идёт по `parentUuid` от последней записи назад к началу. И вот тут ключевой момент: **backtrace обрывается на compact boundary**, потому что `logicalParentUuid` ссылается на UUID, который не существует в текущей ветке (это UUID из записей ДО compact, которые были в предыдущей сессии).

Из лога:
```
Backtrace ended - UUID not found: a9874018-5ab8-4172-b6cc-e688fab794fe
```

4. **Строит Session Tree для отображения** — и показывает дерево сегментов:
```
Session Tree:
  2a05b7b5 (root) — 7 prompts, 2 tools
  4b13df35 (fork) — 2 prompts, 5 tools
  05922cf0 (plan mode) — 2 prompts, 1 tool
```

Записи ДО compact отображаются в дереве как **root-сегмент**, потому что они физически присутствуют в JSONL-файлах. Но **в контекст модели** они не попадают — модель получает только саммари.

---

## Различия: `/compact` vs `/clear`

| Аспект | `/compact` | `/clear` |
|--------|-----------|----------|
| Контекст модели | Очищается, заменяется саммари | Полностью очищается, ничего нет |
| JSONL-файл | Записи остаются, добавляется `compactboundary` + саммари | Контекст сбрасывается |
| Session chain | Продолжает ту же цепочку сессий | Та же сессия, но пустой контекст |
| При `--continue` | Видна вся история в Session Tree (root → fork → plan mode...) | — |
| Знание о прошлом | Есть через саммари | Нет |

- **Ты при `claude -c`** видишь **Session Tree** — это инструмент экспорта/визуализации, который показывает полное дерево сессий из JSONL-файлов. Он специально показывает root-сегмент со старыми промптами, чтобы ты видел историю.
