# Дизайн системы промптирования

*Как устроена архитектура промптов в Noted Terminal — от хранения до момента вставки в терминал. Не "что делает каждый промпт", а как работает сама машинерия.*

---

## Две независимые системы

В приложении два параллельных механизма промптов. Они живут в разных таблицах, отображаются в разных колонках настроек и вызываются разными способами.

### AI промпты (левая колонка)

**аналитические пресеты** *каждый = связка "текст + модель + уровень размышления"*
    - таблица `ai_prompts`: id, name, content, model, thinking_level, color, is_built_in, show_in_context_menu, position
    - модель привязана к промпту: Research работает на Flash, Rewind — на Pro. Пользователь выбирает при создании
    - thinking_level: NONE / LOW / MEDIUM / HIGH — бюджет токенов на размышления (от 0 до 50K)
    - built-in промпты (Research, Compact, Description, Rewind, Adopt Summary) нельзя удалить, но можно редактировать
    - кастомные создаются через `+ Добавить промпт`, получают id = `prompt-<timestamp>`
    - цвет задаётся per-prompt — влияет на карточку в настройках и бейдж в контекстном меню
    - теги на карточках: **FLASH**|**PRO** (по модели) + **ПКМ** (show_in_context_menu) + thinking level если != HIGH

**как вызываются**
    - правый клик по терминалу → пункты верхнего уровня контекстного меню (Research, Compact, ...)
    - при клике → renderer получает `context-menu-command(type='ai-prompt', promptId)`
    - renderer запускает Gemini с конкретной моделью и thinking_level из промпта
    - флаг `show_in_context_menu` управляет видимостью — промпт может существовать только для внутреннего использования (Rewind)

### Текстовые вставки (правая колонка)

**атомарные сниппеты** *чистый текст без привязки к модели — вставляется в поле ввода как есть*
    - таблица `prompts`: id, title, content, position, group_id
    - нет модели, нет thinking_level — это просто текст
    - вставляется буквально: контент промпта → в буфер ввода терминала
    - используются как кирпичики: `docs_search`, `ultrathink`, `опиши user flow` — каждый решает одну задачу

**как вызываются**
    - правый клик по терминалу → `Insert Prompt` → подменю с группами и промптами
    - renderer получает `context-menu-command(type='insert-prompt', content)` — сырой текст
    - текст вставляется в терминал через PTY (без запуска агента, без выбора модели)

## Группировка текстовых вставок

**таблица `prompt_groups`** — id, name, position, description, is_collapsed

### Shared Position Namespace

Группы и негруппированные промпты живут в **едином пространстве позиций**. Поле `position` определяет порядок в плоском списке, позволяя свободно перемежать:

```
position 0: [группа "база"]
position 1: [группа "meta"]
position 2: [промпт "edge cases"]    ← негруппированный
position 3: [группа "объясни"]
position 4: [промпт "тесты и auto/context.md"]  ← негруппированный
```

**Двойное назначение `position` на промптах:**
- `group_id = NULL` → position = место в плоском списке (рядом с группами)
- `group_id != NULL` → position = порядок внутри группы (не участвует в flat ordering)

### Структура в контекстном меню

Группа → submenu. Негруппированный промпт → пункт верхнего уровня внутри `Insert Prompt`:

```
Insert Prompt >
├─ база >
│  ├─ docs_search
│  ├─ ultrathink
│  ├─ проверка после реализации фичи
│  ├─ документация
│  ├─ логи. никаких предложений
│  └─ опиши user flow
├─ meta >
│  ├─ подумай - на основе истории диалога
│  ├─ то что я описал выше - только один из примеров
│  └─ сначала обсуди
├─ объясни >
│  └─ ...
├─ edge cases           ← негруппированный
├─ тесты и auto/context.md  ← негруппированный
└─ externali >
   └─ ...
```

### Collapse Persistence

Состояние сворачивания группы (`is_collapsed`) хранится в БД — переживает перезапуск. В настройках группу можно свернуть кликом по стрелке; в контекстном меню группы всегда раскрыты (submenu).

### Drag-and-Drop (Pointer Events)

Перетаскивание реализовано через Pointer Events, не HTML5 DnD API:
- **Floating Clone**: при зажатии grip-иконки создаётся `position: fixed` клон, следующий за курсором
- **Static Midpoints**: центры карточек фиксируются один раз при начале перетаскивания — все расчёты позиции используют эти замороженные координаты (фикс джиттера от анимированных gap-слотов)
- **Атомарный Drop**: CSS-переходы отключаются синхронно с React-батчем при отпускании — gap исчезает мгновенно

## Системные промпты (нижняя часть левой колонки)

Отделены визуально линией `СИСТЕМНЫЕ ПРОМПТЫ`. Это не промпты для вставки — это конфигурация поведения AI-агентов.

### Claude Default

**стартовый промпт для Claude Code** *инжектируется при запуске сессии*
    - хранение: `app_state` таблица, ключ `claudeDefaultPrompt`
    - тогл вкл/выкл: `localStorage('noted-terminal-claude-default-prompt-enabled')`
    - содержит инструкции типа "прочитай файл docs/context.md ultrathink и четко следуй его инструкциям"
    - при выключенном тогле — промпт существует, но не применяется

### Documentation

**промпт для обновления knowledge-базы** *инжектируется в Update API пайплайн*
    - два режима: **ФАЙЛ** (читает из `/Users/fedor/Global-Templates/.../docs-rules.prompt.md`) или **INLINE** (текст в textarea)
    - хранение: `localStorage('noted-terminal-doc-prompt')` как объект `{ useFile, filePath, inlineContent }`
    - используется в двухстадийном пайплайне: API-анализ → Haiku-применение

### Post-check

**верификационный промпт** *вставляется после основного анализа для ловли слепых пятен*
    - ищет два паттерна: "шрамы формата данных" (места где разработчик получил пустой ответ из-за неожиданного формата) и "засорение CLAUDE.md" (предложения добавить в CLAUDE.md то, что должно быть в knowledge/)
    - тогл: `localStorage('noted-terminal-post-check-docs')`
    - индикатор: фиолетовая точка (6px) в хедере System

## Композиция: "один промпт внутри другого"

Система не поддерживает формальную вложенность промптов (промпт A содержит промпт B). Вместо этого работает **паттерн последовательной сборки** — пользователь-архитектор составляет сложную инструкцию из атомарных кирпичиков:

### Типичный flow

```
1. Insert Prompt > база > docs_search
   → текст "вызови MCP docs_search перед работой..."

2. [пользователь дописывает свою задачу]

3. Insert Prompt > meta > сначала обсуди
   → текст "не начинай реализацию, сначала предложи план..."

4. Enter — всё отправляется одним сообщением
```

Каждый сниппет — один абзац. Вместе они формируют мульти-инструкцию с нужными constraints. Порядок важен: `docs_search` идёт первым (загрузи контекст), потом задача, потом `meta` (как подходить).

### Transclusion (внешняя композиция)

Documentation промпт может ссылаться на файл из `Global-Templates`. Этот файл в свою очередь использует `<!-- @include: ./правила-документации-2.md -->` — трансклюзию, которая подставляет содержимое при рендере. Один источник правды, несколько точек использования.

## Хранение и персистентность

| Что | Где | Формат |
|-----|-----|--------|
| AI промпты | SQLite `ai_prompts` | row per prompt |
| Текстовые вставки | SQLite `prompts` | row per snippet |
| Группы | SQLite `prompt_groups` | row per group |
| Claude Default текст | SQLite `app_state` | key-value |
| Claude Default вкл/выкл | localStorage | boolean |
| Documentation config | localStorage | `{ useFile, filePath, inlineContent }` |
| Post-check текст | localStorage | string |
| Post-check вкл/выкл | localStorage | boolean |
| Auto-apply вкл/выкл | localStorage | boolean |
| Rewind prompt id | SQLite `app_state` | string (prompt id) |

**Почему двойное хранение (SQLite + localStorage)?**
- SQLite — для данных, которые являются контентом (тексты промптов, структура групп). Переживают переустановку, могут бэкапиться.
- localStorage — для UI-preferences (тоглы, последний выбор). Быстрый доступ из renderer без IPC.

## IPC контракт

### Текстовые вставки
```
prompts:get          → { success, data: Prompt[] }
prompts:save         → void (bulk replace)
prompt-groups:get    → { success, data: Group[] }
prompt-groups:save   → void (bulk replace)
```

### AI промпты
```
ai-prompts:get       → { success, data: AIPrompt[] }
ai-prompts:save      → void (upsert single)
ai-prompts:delete    → void (by id, non-built-in only)
```

### Системные настройки
```
app:getState(key)        → value
app:setState({ key, value }) → void
```

## Точки расширения (куда система растёт)

**Текущие ограничения:**
- один уровень вложенности (группа → промпт, но не группа → группа)
- нет формальных ссылок между промптами (промпт A не может include промпт B)
- AI промпты и текстовые вставки — параллельные миры без пересечений

**Направления:**
- группы внутри групп — потребует рекурсивного рендера в контекстном меню и дерева в настройках
- шаблонные переменные в сниппетах (`{{cwd}}`, `{{selection}}`) — превратит вставки из статического текста в генераторы
- объединение двух систем — AI промпт мог бы содержать цепочку текстовых вставок как "рецепт"
