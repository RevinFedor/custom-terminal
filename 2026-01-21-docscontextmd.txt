⏺ Техническая проблема: Мерцание терминала при переключении табов

  Стек технологий

  - Electron + React (renderer process)
  - xterm.js с WebGL addon и FitAddon
  - Терминалы переключаются через CSS display: block/hidden

  Симптом

  При переключении между табами терминала происходит визуальное мерцание —
  экран на мгновение выглядит как будто увеличен на 200%, затем возвращается
   к нормальному масштабу.

  Данные из логов

  [Terminal:-tab-3] safeFit(ResizeObserver) BEFORE: cols=84 rows=48
  container={"w":0,"h":0}
  [Terminal:-tab-3] safeFit(ResizeObserver) AFTER: cols=12 rows=6

  [Terminal:-tab-4] safeFit(ResizeObserver) BEFORE: cols=12 rows=6
  container={"w":645,"h":732}
  [Terminal:-tab-4] safeFit(ResizeObserver) AFTER: cols=84 rows=48

  [Terminal:-tab-4] safeFit(active-useEffect) BEFORE: cols=84 rows=48
  container={"w":645,"h":732}
  [Terminal:-tab-4] safeFit(active-useEffect) AFTER: cols=84 rows=48

  Корень проблемы

  1. Скрытый терминал (CSS display: hidden) имеет container={w:0, h:0}, но
  xterm.js хранит старые размеры cols=84, rows=48
  2. При переключении срабатывает ResizeObserver и вызывает FitAddon.fit():
    - Скрытый терминал: размеры падают с 84x48 → 12x6 (минимальные)
    - Активный терминал: размеры растут с 12x6 → 84x48
  3. WebGL addon перерисовывает буфер при изменении размеров. Переход от
  маленьких размеров (12x6) к большим (84x48) вызывает визуальный артефакт —
   контент сначала рендерится в маленьком масштабе, затем масштабируется.
  4. safeFit вызывается дважды для активного таба:
    - Из ResizeObserver (при появлении элемента)
    - Из useEffect([active]) через 50ms

  Дополнительный контекст

  - devicePixelRatio: 2 (Retina display)
  - Терминалы не уничтожаются при переключении — остаются в DOM со скрытым
  display
  - WebGL рендерер более чувствителен к resize чем canvas fallback

> В тредах r/electronjs и r/reactjs за конец 2025 года эту проблему часто 
называют "xterm.js WebGL accordion effect".

Ты абсолютно точно определил корень зла: xterm.js не должен знать, что его 
контейнер схлопнулся в 0x0. WebGL аддон пытается отрисовать текстуру 
размером 0x0 (или минимальным 12x6), растягивает её на старый CSS-размер 
холста, и ты видишь эти пиксельные "артефакты увеличения", пока 
ResizeObserver не отработает второй раз.

Вот решение, основанное на коллективном опыте 2025-2026 (включая подходы из 
исходников VS Code, который тоже на Electron).

Решение 1: "The Guard Clause" (Самое быстрое)

Самая частая ошибка — позволять FitAddon срабатывать, когда контейнер скрыт.
 Нужно жестко запретить ресайз, если размеры контейнера невалидны.

В твоем хуке/эффекте, где инициализируется ResizeObserver:

code
TypeScript
download
content_copy
expand_less
// Вместо обычного вызова
const resizeObserver = new ResizeObserver((entries) => {
  // 1. Проверка на "скрытость"
  // Если ширина или высота 0 - это display: none. ИГНОРИРУЕМ.
  // Мы НЕ хотим, чтобы xterm сбрасывал буфер до 12x6.
  if (entries[0].contentRect.width === 0 || entries[0].contentRect.height 
=== 0) {
    return;
  }
  
  // 2. Оборачиваем в requestAnimationFrame для защиты от Layout Thrashing
  // (стандарт React 19 best practices для тяжелых рендереров)
  window.requestAnimationFrame(() => {
    fitAddon.fit();
  });
});

resizeObserver.observe(containerRef.current);

Почему это помогает: Терминал остается в памяти с размерами 84x48. Когда ты 
переключаешь таб обратно на display: block, он визуально появляется сразу с 
правильным буфером. ResizeObserver сработает, увидит, что размеры 84x48 (или
 изменились легально), и просто обновит вьюпорт без скачка.

Решение 2: CSS Pattern "Off-screen" (Архитектурно правильное)

На Reddit (пост от maintainer-а одного Electron-терминала) советуют 
отказаться от display: none для WebGL контекстов.

display: none в Chromium (движке Electron) может вызывать Context Loss 
(потерю WebGL контекста) для экономии памяти. Восстановление контекста + 
ресайз = мерцание.

Как делают в 2026:
Используй технику "Visually Hidden". Элемент остается в DOM с размерами, но 
убирается из поля зрения.

code
CSS
download
content_copy
expand_less
/* Вместо .hidden { display: none; } */
.terminal-hidden {
  position: absolute; /* Убираем из потока, чтобы не ломать верстку */
  visibility: hidden; /* Скрываем визуально, но DOM-геометрия сохраняется! 
*/
  z-index: -9999;
  top: 0;
  left: 0;
  /* Важно: не ставить width: 0, пусть сохраняет текущую ширину */
}

.terminal-active {
  position: relative;
  visibility: visible;
  z-index: 1;
}

Плюсы:

ResizeObserver не срабатывает с размерами 0x0, так как при visibility: 
hidden размеры элемента сохраняются.

WebGL контекст не умирает.

Мгновенное переключение без перерисовки.

Решение 3: Pause Rendering (React-специфичное)

Если ты используешь кастомный хук для xterm, добавь паузу рендеринга.

code
TypeScript

content_copy
expand_less
useEffect(() => {
  if (!isActive) {
    // В React 19 / xterm 6.x (условно) можно приостановить рендер
    // В xterm 5.x это делалось через отключение WebGL аддона, но это 
дорого.
    // Просто не вызываем fit().
    return; 
  }

  // FORCE FIT при активации
  // Нужно дать браузеру 1 кадр на пересчет CSS (paint), прежде чем звать 
fit
  const frameId = requestAnimationFrame(() => {
     fitAddon.fit();
  });

  return () => cancelAnimationFrame(frameId);
}, [isActive]);
Итоговый вердикт (Что выбрать)

Сначала примени Решение 1 (Guard Clause). В 90% случаев на Reddit пишут, что
 проверка if (width === 0) return убирает "зум-эффект". Проблема именно в 
том, что xterm сбрасывается в минимум, а WebGL натягивает этот минимум на 
экран.

Если не помогло полностью — Решение 2 (visibility: hidden). Это 
"пуленепробиваемый" вариант для Electron, так как он обходит оптимизации 
Chromium по выгрузке невидимых текстур.

P.S. Убедись, что fitAddon.fit() не вызывается синхронно внутри 
useLayoutEffect при маунте скрытого таба. Это частая причина логов 
container={"w":0,"h":0}.
